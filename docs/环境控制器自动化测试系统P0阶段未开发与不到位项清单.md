# 环境控制器自动化测试系统P0阶段未开发与不到位项清单

> 对照文档：`docs/环境控制器自动化测试系统开发任务表.md`、`docs/环境控制器自动化测试系统开发方案P0.md`。  
> 代码检查日期：2026-06-11。  
> 检查范围：`frontend`、`backend`、`shared`、`scripts`、`docs/ate-deploy.md`、`start-ate.bat`。

## 一、总体结论

当前工程已经补齐了一批 ATE 相关文件，例如 `shared/constants.js`、`backend/ate/AteFrameCodec.js`、`backend/ate/AteTcpClient.js`、`backend/ate/TestManager.js`、`backend/api/test.js`、`frontend/src/views/AteTest.vue` 等；协议单元脚本和前端构建也能通过。

但从 P0 方案和任务表要求看，当前仍偏“半成品框架 + 前端演示工作台”，还没有形成可用于产线的真实闭环。核心问题是：

1. 后端测试主流程没有真正接入 `AteTcpClient` 的 9001 TCP+JSON 协议链路，实际主要走 Modbus `0x8000` 寄存器写读。
2. 前端 `AteTest.vue` 仍保留大量本地 mock 执行器、随机结果、模拟报文和硬编码项目树。
3. 报告、配置、手动点检、辅助工装板、业务逻辑测试、真机/模拟器端到端联调仍未完整闭环。
4. 部署口径、端口、静态前端托管和 WebSocket 消息类型存在不一致，实际落地时容易“构建能过、现场打不开或收不到状态”。

## 三、P0 必须补齐项

### 1. ATE TCP+JSON 主链路没有接入 TestManager

任务表要求后端完成 `AteTcpClient`、`AteMessageRouter`、`TestManager` 的真实联动，覆盖 `heartbeat`、`test.enter`、`test.exit`、`control.force_io`、`report`、ACK/error。

当前问题：

- `AteTcpClient.js` 和 `AteMessageRouter.js` 文件存在，但 `server.js` / `TestManager.js` 没有实例化和注入使用。
- `TestManager._enterTestMode()` 注释写“通过 Modbus TCP 发送 test.enter”，实际只是写 `BLOCK_TEST_STATUS.START = TEST_CMD.START`。
- `_safeRelease()` 注释写发送 `test.exit`，实际写的是 Modbus `RESET`。
- `config.write`、`properties.get`、固件主动 `report` 没有进入测试主流程。
- **新增缺陷**：`TestManager` 的 500ms Modbus 状态轮询定时器缺乏超时和断线判定。如果测试中下位机断电，后端只会持续报错输出 `console.error`，自检状态将永久卡死在“进行中”而无法自动判断为“超时”或“连接异常”。
- **新增缺陷**：测试管理未遵循 `debug_issues.md` 中对于下位机 Modbus TCP 通信 12 秒重连冷却时间的约束。当频繁触发 `resetSession` 或多次断开重连时，未建立连接冷却保护，极易导致固件 Modbus 引擎锁死。

影响：无法验证固件 9001 TCP+JSON 协议，P0 第 4、5、17 章的核心链路未闭合，且异常断电/断线时系统无法闭环收尾。

建议补齐：

- 在后端建立按设备管理的 `AteTcpClient` 实例池。
- `TestManager` 启动测试时先 `test.enter`，配置走 `config.write`，手动点检走 `control.force_io`，停止/异常走 `test.exit`。
- 固件 `report` 转换为 `test_progress_update`、`ate_device_log` 或状态快照。
- 在 `_pollTestStatus` 中加入连续失败计数器。当 Modbus 或 TCP 读取连续失败达到阈值（如 5 次）时，自动触发会话异常中止（`session.state = SESSION_STATE.ERROR`），并执行 `_safeRelease`。
- 在 `AteClientPool` 和 Modbus 重连逻辑中引入冷却定时器，限制高频重复连接行为。

### 2. 前端 ATE 页面仍是本地 mock 驱动


任务表 6.1/6.2 要求 `AteTest.vue` 从 mock 改为 Pinia 真实 session 驱动。

当前问题：

- 页面仍有 `mockInterval`、`startMockSerial()`、`updateRegisterAssertMock()`、`executeStep()`。
- 无后端连接时使用 `Math.random()` 生成 PASS/FAIL。
- 有 WebSocket 时也只是每个节点调用一次 `deviceStore.startTest()`，然后用本地 `setInterval` 轮询 `deviceStore.ateStatus` 判定当前行。
- 左侧 `testTree` 是组件内硬编码的 19 个代表项，没有从 `TestCatalog` 或后端接口获取。
- 当前项目树缺少第 9 项基础自检“RS485 热切换”，并与后端 9 项基础目录不一致。

影响：UI 显示结果可能与真实底板测试状态不一致，存在“前端显示合格但后端/固件未完成”的风险。

建议补齐：

- 移除本地测试执行器和随机 PASS/FAIL。
- `AteTest.vue` 只根据 `deviceStore.ateSession / ateTimeline / ateSummary` 渲染。
- 后端提供测试目录接口，前端项目树从同一数据源加载。
- 刷新页面时通过 `get_test_session` 恢复当前会话。

### 3. WebSocket 消息类型存在重复和不一致

当前问题：

- `TestManager._emitSessionUpdate()` 会广播 `type: test_${eventType}`，例如 `test_test_finished`。
- `server.js` 又监听 `testManager.on('test_finished')` 并调用 `pushTestFinished()`，推送 `test_finished_notification`。
- `deviceStore.js` 用 `msg.type.startsWith('test_')` 兜底处理，并特别判断 `test_test_finished`、`test_test_error`。

影响：前端可能收到两套不同结构的测试消息；后续对接真实 UI、报告下载和错误处理时容易出现状态重复、丢失或误判。

建议补齐：

- 统一后端只推任务表定义的标准消息：`test_progress_update`、`test_finished_notification`、`test_error`、`ate_device_log`、`ate_raw_frame`。
- `TestManager` 只 emit 业务事件，由 `server.js` 或 `WebSocketMgr` 统一转换 WebSocket 消息。

### 4. 报告 API 与报告生成未完全可用

当前问题：

- `backend/api/test.js` 中 `/reports/:filename` 路由写在 `/reports/:filename.json` 和 `/reports/:filename.html` 前面，会优先匹配带扩展名的请求。
- `getReport(fileName)` 只按 JSON 解析文件，因此请求 `.html` 文件时会解析失败并返回空。
- `/reports/:filename.html` 构造 `htmlFilename` 后读取 HTML 文件，但仍调用 `getReport()`，同样无法解析 HTML。
- `TestReportService._generateJsonReport()` 里计算 `testMask` 的方式调用空对象上的 `TestCatalog.prototype.getItemById`，结果基本会是 `0`。
- 报告缺少系统日志、环控器日志、原始 TCP/Modbus 报文摘要和参数快照细节。
- **新增缺陷**：`TestReportService` 在生成报告时试图从 `TestSession` 读取设备序列号 `deviceSn` 和固件版本 `firmwareVersion`。但在 `TestManager.js` 的 `TestSession` 类中根本没有获取并保存这两个字段，导致最终生成的 JSON 报告和 HTML 报告中这些核心质量追溯字段永久为空。

影响：报告可追溯性不足，质量追溯的核心 SN/版本数据丢失，HTML 下载接口可能不可用。

建议补齐：

- 报告 API 改成显式读取 JSON 原始报告，再按需生成 HTML 响应。
- 修正路由顺序或改为 `/reports/json/:id`、`/reports/html/:id`。
- `TestReportService` 持有真实 `TestCatalog` 实例，正确计算 `testMask`。
- 将 timeline、寄存器快照、错误码说明、系统日志、设备日志、原始通信摘要落盘。
- 在 `TestSession` 初始化或测试开始时，通过 `properties.get` 请求设备物模型的 `device.sn` 与固件版本，将其保存到 `TestSession` 中。或者在启动测试时允许操作员通过前端输入设备 SN 并传递给后端。

### 5. ATE 寄存器定义存在 8/9 项和地址重叠问题

当前问题：

- `ATE_MASK_ALL = 0x01FF` 表示 9 项基础自检，但 `SINGLE_RESULT_BASE` 注释和解析只覆盖 `0x8010-0x8017` 共 8 个寄存器。
- `ERROR_CODE_BASE` 同样只解析 8 个错误码。
- `TestProtocol.parseSingleResults()` 固定循环 8 项。
- 模拟器 `singleResults`、`errorCodes` 也都是 8 项数组。
- `BLOCK_TEST_STATUS.SESSION_ID = 0x8005` 被当作 `0x8005-0x8006` 两个寄存器解析，而 `TEST_MASK` 又定义为 `0x8006`，存在重叠。
- `DIAG_CHANNEL/EXPECTED/ACTUAL` 与 `ERROR_CODE_BASE + 1/2/3` 地址重叠。

影响：第 9 项 RS485 热切换无法完整记录结果/错误；会话 ID、测试掩码和诊断字段可能互相覆盖。

建议补齐：

- 按 P0 和固件真实实现重新确认 `0x8000` 区地址表。
- 明确 9 项结果区和错误码区至少覆盖 9 个寄存器。
- 修正 `shared/constants.js`、`TestProtocol.js`、模拟器和文档中的地址定义。

### 6. 手动强制 IO 尚未实现

当前问题：

- `server.js` 对 `manual_force_io_request` 只有 TODO，并直接返回成功。
- `AteTest.vue` 手动点检页只显示 8 路继电器开关，没有绑定 `deviceStore.manualForceIo()`。
- 任务要求支持 22 路继电器/AO 输出、安全超时释放和 `test.enter/test.exit` 接管。

影响：手动点检无法真实控制底板，且 UI 可能误导操作员以为已执行。

建议补齐：

- 后端用 `AteTcpClient.forceIo()` 或 Modbus 强制输出实现真实控制。
- 前端扩展到 22 路继电器、AO 输出和超时时间。
- UI 显示强制接管、释放结果和失败原因。

### 7. 部署和端口口径不统一，静态前端未托管

当前问题：

- `deviceStore.js` WebSocket 写死 `ws://${window.location.hostname}:3001`。
- `frontend/vite.config.js` 代理目标是 `http://localhost:3001`。
- `backend/server.js` 默认端口是 `3000`，但 `backend/config/devices.json` 配置为 `3001`。
- `start-ate.bat` 构建了 `frontend/dist`，但后端只托管 `backend/public`，没有托管 `frontend/dist`。
- `start-ate.bat` 打开 `http://localhost:3001`，如果没有单独静态服务器，可能只看到后端 API，而不是前端应用。

影响：车间部署时可能出现后端已启动、浏览器却打不开前端页面或 WebSocket 连错端口。

建议补齐：

- 统一端口来源，前端 WebSocket 使用当前页面 origin 或环境变量。
- 后端生产模式托管 `frontend/dist`，并将 `/ate` 回退到 `index.html`。
- `start-ate.bat` 启动后端后打开 `/ate`，并检查端口占用。

### 8. 业务逻辑测试目录和 P0 第 14 章差距较大

当前问题：

- `TestCatalog.js` 只有 9 项基础自检和 5 个代表性业务项。
- P0 第 14 章要求的小窗、滑窗、幕帘、风机、水帘、喷淋、加热、照明、异常健壮性等没有完整目录和判定公式。
- 业务项没有 `mask`，传入 `TestManager` 后不会形成有效 `testMask`。
- 前端的 19 个项目与后端目录不是同一来源。

影响：第一版 9 项基础自检以外的业务闭环基本不可用，且前后端项目 ID 容易错位。

建议补齐：

- 后端 `TestCatalog` 定义完整测试树，包括执行方式、依赖、超时、判定公式、错误说明。
- 前端从后端目录加载，不再硬编码私有项目。
- 明确哪些业务项走 TCP JSON、哪些走 Modbus 回读、哪些需要辅助板。

建议补齐：统一为浅灰工具型风格，去掉 emoji 功能图标；若继续用 Font Awesome，需要明确这是项目约定并保持离线可用。

## 八、具体开发细节建议

本节按“可以直接拆开发任务”的粒度补充，建议优先从 8.1 到 8.6 顺序推进。每完成一个小闭环，都先跑本地验证，再进入下一项。

### 8.1 统一 ATE 寄存器地址表

目标：先把协议地基修正，避免后端、模拟器、报告和前端全部围绕错误地址继续开发。

建议修改文件：

| 文件 | 修改点 |
| :--- | :--- |
| `shared/constants.js` | 重新定义 `BLOCK_TEST_STATUS`，明确 9 项结果区、9 项错误码区、诊断区、会话 ID 和测试掩码不重叠。 |
| `backend/ate/TestProtocol.js` | `parseSingleResults()`、`parseErrorCodes()` 改为解析 9 项。 |
| `backend/ate/TestManager.js` | 轮询 `0x8000` 区时按新地址读取和更新 timeline。 |
| `backend/ate/mock/AteDeviceSimulator.js` | 模拟寄存器数组改为 9 项结果和 9 项错误码。 |
| `scripts/test-ate-protocol.js` | 增加第 9 项 RS485 热切换结果和错误码断言。 |

建议落地规则：

- `ATE_MASK_ALL` 保持 `0x01FF`。
- 结果区必须能保存 item 1~9 的状态。
- 错误码区必须能保存 item 1~9 的错误码。
- `SESSION_ID` 若使用 32 位，应占用两个连续寄存器，后续字段从下一个地址开始。
- 诊断区不要复用错误码区地址。

验收标准：

- `node scripts/test-ate-protocol.js` 通过。
- 测试脚本能断言第 9 项结果和错误码。
- 文档、常量、解析器、模拟器中的 9 项数量一致。

### 8.2 打通后端 TCP+JSON 最小闭环

目标：让 `TestManager` 真正通过 `AteTcpClient` 进入/退出测试模式，并接收固件 ACK/report。

建议新增或调整：

| 文件 | 具体开发点 |
| :--- | :--- |
| `backend/ate/AteClientPool.js` | 新增设备级 ATE TCP 客户端池，按 `deviceKey` 复用 `AteTcpClient`。 |
| `backend/ate/TestManager.js` | 构造函数注入 `ateClientPool`；`_enterTestMode()` 调用 `client.enterTestMode()`；`_safeRelease()` 调用 `client.exitTestMode()`。 |
| `backend/server.js` | 初始化 `AteClientPool`，并传入 `TestManager`。 |
| `backend/WebSocketMgr.js` | 用标准消息推送 TCP 原始帧、设备 report 和错误。 |

建议接口形态：

```js
// backend/ate/AteClientPool.js
class AteClientPool {
  constructor({ devices, wsManager, options }) {}
  getClient(deviceKey) {}
  async connect(deviceKey) {}
  async disconnect(deviceKey) {}
  getStatus(deviceKey) {}
}
```

`TestManager` 建议流程：

1. `startSession()` 创建 session。
2. `ateClientPool.connect(deviceKey)` 连接 `deviceIp:9001`。
3. `client.enterTestMode()` 进入测试模式。
4. `client.writeConfig()` 下发基础参数。
5. Modbus 写 `TEST_MASK` 和 `START`，触发固件自检。
6. 固件主动 `report` 或 Modbus 轮询更新进度。
7. 完成/失败/停止时 `client.exitTestMode()`。

验收标准：

- 断开 9001 时，前端收到 `test_error`，错误文案明确“ATE TCP 9001 不可达”。
- `test.enter` ACK 超时会中止测试并安全释放。
- 停止测试时，无论 Modbus 是否正常，都尝试发送 `test.exit`。

### 8.6 实现手动强制 IO

目标：手动点检页能真实控制 22 路继电器/AO，并具备超时释放。

建议修改：

| 文件 | 开发细节 |
| :--- | :--- |
