## 1、功能定位与概述

### 1.1 背景与设计动机

在猪舍环控系统中，通风量的精确控制直接关系到舍内温度、湿度、氨气浓度等关键环境指标。不同季节、不同生长阶段的猪只需要不同的通风量等级。当通风等级较低时（如冬季最小通风），不需要所有风机同时全速运行——此时如果让所有风机都以低频率持续运转，既浪费电能，又容易造成局部风速过大、温度波动等问题。

为解决这一矛盾，系统引入了三种风机控制模式：

- **间歇控制**：让单台风机周期性地"开-停"交替运行，在保持平均通风量的同时降低瞬时功率，同时给猪舍一个"呼吸"节奏，避免持续吹风导致体感温度过低。
- **轮替控制**：在多台风机的场景下，让它们"排队"依次运行，同一时刻只有一台工作。这样做既能保证总通风量达标，又能让每台风机的运行时间均匀分配，延长设备寿命，也方便在轮替间隙发现故障风机。
- **关联间歇**：这是微正压通风拓扑下的特殊需求。排风风机间歇停止时，猪舍内正压会降低；此时需要送风侧做出相应调整（降频或切换风机类型），以维持"先送后抽、先减抽后减送"的微正压节奏。

这三种模式并非互斥的全局开关，而是**按单台风机粒度配置**的。同一个通风等级下，送风侧的1号变频风机可能配置为"间歇"，2号变频风机配置为"全速轮替"，而送风定速风机配置为"间歇关联"。这种灵活性使得现场工程师可以根据猪舍的实际布局和风机特性，精细调整每一级的通风策略。

### 1.2 三种模式定义

通风系统中，风机的工作模式由逻辑表 `logicEnControl[]`（送风侧）和 `logicAirPumping[]`（抽风侧）中的 `fc_workMode` / `cs_workMode` 字段决定。定义于 [database.h:42-49](bsp/stm32/stm32f407-atk-explorer/applications/middleware/storage/database.h#L42-L49)：

| 值 | 枚举名 | 含义 | 控制器 |
|:---|:---|:---|:---|
| 0 | `interval` | 间歇通风 | `ventilation_fan_executor` |
| 1 | `trensRun` | 全时开启 | 直接输出 |
| 2 | `Fanclose` | 关闭 | 直接输出 |
| 3 | `takeTurns` | 全速轮替 | `ventilation_rotation_controller` |
| 4 | `takeBreak` | 间歇轮替 | `ventilation_rotation_controller` |
| 5 | `linkBreak` | 间歇关联 | `ventilation_fan_relate_controller` |

### 1.2 模式适用场景

| 模式 | 典型场景 | 说明 |
|:---|:---|:---|
| 间歇 (`interval`) | 低通风等级下单组风机的周期性开停 | 一组风机按 opTime/clTime 周期切换，适用于变频或定速风机 |
| 全速轮替 (`takeTurns`) | 多台风机依次全速运行 | 多台风机逐个开启，每台运行 opTime 后关闭并切换到下一台，同一时刻只有一台运行 |
| 间歇轮替 (`takeBreak`) | 多台风机依次间歇运行 | 多台风机逐个开启，每台运行 opTime 后关闭，等待 clTime 后切换到下一台 |
| 间歇关联 (`linkBreak`) | 排风间歇时联动送风 | 排风风机处于间歇周期时，送风风机按关联规则同步开停以维持微正压 |

### 1.3 关键数据结构

> 共享参数 `logicEnControl[]` 的完整定义详见 [负压控制逻辑](负压控制逻辑.md) §2.1，此处仅列出间歇/轮替/关联间歇控制相关的数据结构。

间歇控制的核心是一个简单的双态状态机（开启/关闭），由一个全局结构体数组驱动。每台风机在进入间歇模式时，会将自己的参数（开多久、关多久、频率多少、哪一组风机）写入对应槽位，然后由 1 秒定时器统一驱动所有槽位的状态跳转。这种"参数注入 + 定时器驱动"的设计，使得间歇逻辑与通风级别计算完全解耦——级别控制器只负责"告诉"间歇模块当前应该用什么参数，间歇模块自己按秒计时并切换开停。

## 2、间歇控制（interval）详解

### 2.1 工作原理概述

间歇控制的本质是一个**以秒为单位的双稳态定时器**。风机在"开启"和"关闭"两个状态之间反复切换，每个状态持续的时间由逻辑表中的 `opTime`（开启时长）和 `clTime`（关闭时长）决定。

例如，当 `opTime=280秒, clTime=20秒` 时，风机每运行 4 分 40 秒后停止 20 秒，形成一个 5 分钟的周期。在这个周期内，风机的"平均通风量"约为全速运行时的 280/300 = 93.3%。通过调整 opTime 和 clTime 的比例，可以精确控制平均通风量——clTime 越长，平均通风量越低，适合冬季最小通风场景。

间歇控制同时支持**变频风机**（PWM 输出）和**定速风机**（继电器输出）。变频风机在开启期间以指定频率运行，关闭期间 PWM 输出归零；定速风机在开启期间继电器吸合，关闭期间继电器释放。两者的状态机逻辑完全一致，只是物理输出接口不同。

### 2.2 触发入口

间歇模式的启动不是由定时器主动发起的，而是由**通风级别执行函数被动注入**的。每当通风系统根据温度计算出新的目标等级并执行时，`ventilation_fan_executor_run_pwm_level()` 或 `ventilation_fan_executor_run_fixed_level()` 会读取该风机在当前等级下的 `workMode`。如果 `workMode == interval`（值为0），就将间歇参数（opTime、clTime、freq 等）写入 `s_interval_info` 对应槽位，并将 `state` 设为 1，正式启动间歇循环。

这意味着间歇模式的生命周期与通风等级绑定：等级切换时，如果新等级的 workMode 不是 interval，旧的间歇状态会被清除（`state` 重置为 0）；如果新等级仍然是 interval，则重新注入新参数并重启。

**PWM 变频风机入口** — [ventilation_fan_executor.c:325-367](bsp/stm32/stm32f407-atk-explorer/applications/app/ventilation/ventilation_fan_executor.c#L325-L367)：

### 2.3 驱动机制

间歇控制的运行不依赖通风主循环线程，而是由一个**独立的 1 秒周期定时器**驱动。这样设计的原因是：通风主循环的执行周期不固定（受温度计算、级别切换、补偿逻辑等影响，可能几秒甚至几十秒才循环一次），而间歇控制需要精确的秒级计时。定时器在系统启动时注册，此后每秒触发一次回调，遍历送风和排风两个间歇槽位，分别执行状态跳转。

定时器在 `ventilation_control_loop.c` 中注册：

[ventilation_control_loop.c:38-47](bsp/stm32/stm32f407-atk-explorer/applications/app/ventilation/ventilation_control_loop.c#L38-L47)：

```c
static void ventilation_control_loop_timer_callback(void* parameter)
{
    if (interval_ventilation_service_can_tick() == RT_TRUE) {
        ventilation_fan_executor_tick_supply_interval();   // 送风间歇 tick
        ventilation_fan_executor_tick_exhaust_interval();  // 排风间歇 tick
    }
    // ...
}
```

**tick 前置条件**（[interval_ventilation_service.c:14-25](bsp/stm32/stm32f407-atk-explorer/applications/middleware/control/interval_ventilation_service.c#L14-L25)）：

定时器回调在执行间歇 tick 之前，会先检查两个全局条件。这两个条件的作用是：当系统处于非自动控制状态时（如远程平台接管了风机控制，或者猪舍处于空栏状态），暂停间歇计时但不清除间歇状态。这样当条件恢复后，间歇可以从暂停时的状态继续运行，而不是从头开始。

| 条件 | 状态 | 结果 |
|:---|:---|:---|
| 远程接管 | `control_mode_service_is_remote_takeover() == RT_TRUE` | 禁止 tick |
| 猪舍为空 | `state_service_is_house_empty() == RT_TRUE` | 禁止 tick |
| 其他 | - | 允许 tick |

### 2.4 送风间歇状态机运行过程

`ventilation_fan_executor_tick_supply_interval()` — [ventilation_fan_executor.c:56-190](bsp/stm32/stm32f407-atk-explorer/applications/app/ventilation/ventilation_fan_executor.c#L56-L190)

送风间歇是整个间歇控制的核心实现。状态机共有三个状态：state=0（空闲）、state=1（开启周期）、state=2（关闭周期）。状态机的驱动完全由 1 秒定时器回调完成——每次回调时，函数检查当前 state 值，执行对应的 tick 累加和条件判断。

在 state=1（开启周期）中，风机正在以逻辑表指定的频率（变频）或继电器吸合（定速）方式运行。每秒 tick 自增 1，当 tick 累计达到 opTime 时，表示"开启时间已用完"，此时执行关闭动作：变频风机的 PWM 输出写 0，定速风机的继电器拉低。随后 state 跳转到 2，tick 归零，进入关闭周期。

在 state=2（关闭周期）中，风机处于停止状态。每秒 tick 自增 1，当 tick 累计达到 clTime 时，表示"关闭时间已用完"，此时执行开启动作：变频风机的 PWM 恢复到间歇启动时保存的 freq 值，定速风机的继电器拉高。随后 state 跳转回 1，tick 归零，重新进入开启周期。

如此循环往复，直到通风级别变化导致 workMode 改变，state 被重置为 0，间歇循环结束。

需要注意的一个细节是"校准保护"：当水帘或幕帘正在执行位置校准（如全开/全关校准）且校准时间已超过 2 分钟时，系统会跳过间歇的关闭动作（通过 `goto noOff` 直接跳转到 state=2 的 tick 累加逻辑）。这是为了防止间歇关闭中断校准过程，导致校准失败或机构卡死。

### 2.5 排风间歇状态机运行过程

`ventilation_fan_executor_tick_exhaust_interval()` — [ventilation_fan_executor.c:195-267](bsp/stm32/stm32f407-atk-explorer/applications/app/ventilation/ventilation_fan_executor.c#L195-L267)

排风间歇的状态机逻辑与送风间歇完全对称——同样的三态结构（空闲→开启→关闭→开启...），同样的 tick 累加和 opTime/clTime 比较。但排风间歇有一个关键的附加职责：**在状态切换的瞬间，通知关联间歇控制器做出联动响应**。

这个附加职责的物理背景是：在微正压猪舍中，排风风机是"抽气"侧，排风停止会导致舍内正压升高（因为送风还在继续），排风恢复会导致正压回落。为了维持稳定的微正压环境，送风侧需要在排风状态变化时做出相应的调整——排风关闭时送风适当降频，排风恢复时送风恢复原频率。这就是关联间歇控制器 `ventilation_fan_relate_controller_apply()` 被调用的时机。

排风间歇还有一个与送风间歇不同的地方：它在关闭和开启时都会检查"补压状态"。当系统正在执行压差补压（`BCdiffLevel != 0`）时，跳过关联间歇的触发。这是因为补压逻辑本身已经在临时提升送风等级，此时如果再叠加关联间歇的调整，两者会互相干扰，导致送风频率忽高忽低。

```
state=1 (开启周期) → tick >= opTime → 关闭排风机
    │
    ├─ 检查是否处于补压状态 (Exe_Ventilationlevel == airPump_Vlevel && BCdiffLevel == 0)
    │  └─ 非补压时：调用 ventilation_fan_relate_controller_apply(STATE_OFF, 0)
    │     → 通知关联间歇控制器：排风已关闭
    │
    ▼
state=2 (关闭周期) → tick >= clTime → 开启排风机
    │
    ├─ 检查是否处于补压状态
    │  └─ 非补压时：调用 ventilation_fan_relate_controller_apply(STATE_ON, 0)
    │     → 通知关联间歇控制器：排风已开启
    │
    ▼
(循环)
```

### 2.6 间歇控制中的安全保护

间歇控制虽然逻辑简单，但在实际猪舍环境中运行时，需要处理多种异常和边界情况。以下是系统中内置的六重保护机制：

**校准保护**（state=1→2 转换时）：猪舍中的水帘和幕帘机构在首次使用或长时间停用后，需要执行位置校准（全开→全关→记录行程时间）。如果校准过程中间歇控制恰好触发关闭，会中断校准流程，导致机构位置丢失。因此，当检测到水帘/幕帘正在校准且校准时间已超过 2 分钟时，间歇的关闭动作被跳过（通过 `goto noOff` 直接进入 state=2 的 tick 累加阶段，但不实际关闭风机）。这样风机继续保持运行，直到校准完成后再正常执行间歇切换。

**补压保护**（间歇启动入口）：当系统处于压差补压状态时（`BCdiffLevel` 不为 0），间歇模式不会被启动。这是因为补压本身就是一种临时提升送风等级的操作，如果此时再叠加间歇的开停周期，会导致送风量剧烈波动，破坏补压效果。具体判断条件是 `ctrLevel > Exe_Ventilationlevel`——当执行的等级高于当前主执行等级时，说明是补压触发的额外执行，不应启动间歇。

**驻极体检测**（state=2→1 转换时，仅定速风机）：驻极体传感器是检测风机是否正常运转的反馈装置。当间歇关闭周期结束、准备重新开启定速风机时，系统会先检查该风机对应的驻极体传感器是否在线（`getbit(App_Run.ElcEnabelBit, ...)`）。如果驻极体断开（传感器故障或风机断线），则不执行开启动作，继电器保持低电平。这可以防止在风机故障状态下反复尝试启动，避免继电器和电机的无效磨损。

**PWM 自动保护位**（state=1→2 和 2→1 转换时）：系统中有一个全局的"自动保护位"机制，用于协调手动控制和自动控制的优先级。当 `fan_semantic_get_main_pwm_auto_guard_bit()` 被置位时（通常意味着 HMI 侧正在手动操作该风机），间歇控制的 PWM 写入操作被跳过。这确保了手动控制的绝对优先权——运维人员在 HMI 上手动调整风机频率时，间歇逻辑不会自动覆盖手动设定值。

**远程接管**（tick 前置检查）：当远程平台（如物联网云平台）通过 MQTT 协议下发了风机控制指令时，系统进入"远程接管"模式。此时间歇定时器的 tick 被完全禁止——既不计时也不切换状态。风机保持在接管前一刻的状态，直到远程控制结束。这避免了远程手动调试时，间歇逻辑自动切换风机状态导致的冲突。

**默认时间保护**（间歇启动时）：逻辑表中的 opTime 和 clTime 由现场工程师通过 HMI 配置。如果配置失误导致两个值都为 0，间歇会陷入"开启后立即关闭、关闭后立即开启"的高频振荡，对继电器和电机造成严重损害。因此系统在间歇启动时检测这种情况，自动替换为默认值 `opTime=280秒, clTime=20秒`，确保间歇周期不低于 5 分钟。

### 2.6 运行时序示例

以变频风机间歇为例，假设配置 `opTime=280秒, clTime=20秒, freq=50%`：

## 3、轮替控制详解

### 3.1 轮替控制的设计背景

在较大规模的猪舍中，单台风机的通风量往往不足以满足高等级通风需求，因此通常会安装多台风机并联工作。但如果让所有风机同时运行，低等级时每台风机只需很小的输出，效率很低；而且长时间低负载运行容易导致电机过热、轴承磨损不均等问题。

轮替控制的核心思想是：**将总通风需求"分配"给多台风机，让它们轮流工作**。比如有 3 台定速风机配置为轮替模式，opTime=120秒，那么每台风机依次运行 2 分钟后关闭，切换到下一台。这样每台风机的实际工作时间是总时间的 1/3，既满足了通风需求，又大幅降低了单机负荷。

轮替与间歇的根本区别在于：间歇是"一台风机自己开开关关"，轮替是"多台风机排队轮流"。间歇只需要一个定时器槽位，轮替需要一个游标（`FixedtakeIndex`）在多台风机之间移动。

### 3.2 全速轮替（takeTurns）

全速轮替是轮替控制的基础形态。它的运行规则非常简单：**按风机编号顺序，每台运行 opTime 后立即切换到下一台，同一时刻只有一台风机在工作**。"全速"的含义是每台风机在运行期间以逻辑表指定的频率（变频）或全功率（定速）输出，不做任何降频处理。

全速轮替适用于以下场景：
- 中等通风等级，需要持续的通风量但不需要所有风机同时运行
- 风机数量较多，希望均衡磨损以延长设备寿命
- 变频风机在低频率下效率较低，不如让单台风机以较高频率运行

**触发条件**：逻辑表中某台风机的 `workMode == takeTurns`（值为3）。

**重要前置条件**：参与轮替的风机数量必须 >= 2 台。如果只有 1 台配置了轮替模式，轮替逻辑不会启动（`TurnNum < 2` 直接返回）。这个限制是合理的——1 台风机没有"轮替"的对象，轮替就退化成了全时开启。

## 4、关联间歇控制（linkBreak）详解

### 4.1 功能目的

关联间歇的核心目的是：**当排风风机处于间歇关闭周期时，联动调整送风风机的输出，以维持猪舍微正压**。

排风间歇关闭 → 猪舍内气压降低 → 需要送风侧做出响应 → 关联间歇控制器介入。

### 4.2 触发时机

关联间歇由排风间歇 tick 函数触发：

1. **排风间歇进入关闭周期**（state=1→2 转换时）：调用 `ventilation_fan_relate_controller_apply(STATE_OFF, 0)`
2. **排风间歇退出关闭周期**（state=2→1 转换时）：调用 `ventilation_fan_relate_controller_apply(STATE_ON, 0)`

触发条件：
- 不处于补压状态（`Exe_Ventilationlevel == airPump_Vlevel && BCdiffLevel == 0`）

### 4.3 关联模式识别

`ventilation_fan_relate_controller_apply()` — [ventilation_fan_relate_controller.c:27-172](bsp/stm32/stm32f407-atk-explorer/applications/app/ventilation/ventilation_fan_relate_controller.c#L27-L172)

控制器首先扫描当前通风等级下所有风机的 workMode，识别哪些风机配置了 `linkBreak` 模式：

```c
FanRelateCtr.CtrMode = 0;
for (int i = 10; i >= 0; i--) {
    if (i == 10) {
        // 检查变频风机
        if (logicEnControl[level].fc_workMode == linkBreak) {
            FanRelateCtr.CtrMode = 1;       // 送风变频关联
            FanRelateCtr.InFanNum[0] = 1;   // 1号变频风机组
        } else if (logicEnControl[level].fc_workMode2 == linkBreak) {
            FanRelateCtr.CtrMode = 1;
            FanRelateCtr.InFanNum[0] = 2;   // 2号变频风机组
        }
    } else {
        // 检查定速风机
        if (cs_workmode[i] == linkBreak) {
            if (FanRelateCtr.CtrMode) {
                FanRelateCtr.CtrMode = 3;   // 变频+定速混合关联
            } else {
                FanRelateCtr.CtrMode = 2;   // 送风定速关联
            }
            FanRelateCtr.InFanNum[1] = i;
        }
    }
}
```

**CtrMode 识别结果**：

| CtrMode | 含义 | 触发条件 |
|:---|:---|:---|
| 0 | 无关联风机 | 所有风机都没有 linkBreak 模式 |
| 1 | 送风变频关联 | 仅变频风机配置了 linkBreak |
| 2 | 送风定速关联 | 仅定速风机配置了 linkBreak |
| 3 | 变频+定速混合关联 | 变频和定速风机都配置了 linkBreak |
| 4 | 特殊模式 | 由 `ctrType==1` 触发（温度变化时的特殊处理） |

### 4.4 各关联模式的控制动作

#### CtrMode=1：送风变频关联

排风间歇关闭时，送风变频风机从 `fc_runFreq[1]` 切换到 `fc_runFreq[0]`（降频）；排风间歇开启时，恢复 `fc_runFreq[1]`。

```
排风间歇 state=1 (开) → 送风变频 = fc_runFreq[1] (较高频率)
排风间歇 state=2 (关) → 送风变频 = fc_runFreq[0] (较低频率)
```

代码摘录：
```c
if (FanRelateCtr.CtrMode == 1) {
    if (ctrValue == STATE_ON) {
        // 排风开启 → 送风变频升频
        ao_output_service_write(AO_PWM, logicEnControl[level].fc_runFreq[1]);
    } else {
        // 排风关闭 → 送风变频降频
        ao_output_service_write(AO_PWM, logicEnControl[level].fc_runFreq[0]);
    }
}
```

#### CtrMode=2：送风定速关联

排风间歇关闭时，送风定速风机开启；排风间歇开启时，送风定速风机关闭。

```
排风间歇 state=1 (开) → 送风定速 = OFF (PIN_LOW)
排风间歇 state=2 (关) → 送风定速 = ON  (PIN_HIGH)
```

#### CtrMode=3：变频+定速混合关联

排风间歇关闭时：开启送风定速，关闭送风变频。
排风间歇开启时：关闭送风定速，开启送风变频。

```
排风间歇 state=1 (开):
    送风定速 = OFF
    送风变频 = fc_runFreq[0] (运行)

排风间歇 state=2 (关):
    送风定速 = ON
    送风变频 = 0 (关闭)
```

这种模式的设计意图是：排风关闭时用定速风机维持基本通风量，排风开启时切回变频精确控制。

#### CtrMode=4：特殊模式

由温度变化触发（`ctrType==1`），用于通风等级变化时的优先处理：

```c
if ((ctrType == 1) && (FanRelateCtr.CtrMode != 0)) {
    FanRelateCtr.CtrMode = 4;
}
```

CtrMode=4 的行为：
- 开启送风定速风机
- 开启送风变频风机到当前等级的 `fc_runFreq[0]`
- 如果有压差补压等级 (`BCdiffLevel`)，使用补压等级的频率

### 4.5 关联间歇与补压的关系

在排风间歇 tick 函数中，关联间歇的触发有补压状态检查：

```c
// ventilation_fan_executor.c 排风间歇 tick
if ((App_Run.Exe_Ventilationlevel == App_Run.airPump_Vlevel) &&
    (App_Run.BCdiffLevel == 0)) {  // 补压时候不执行关联间歇
    ventilation_fan_relate_controller_apply(STATE_OFF, 0);
}
```

**含义**：当系统正在执行压差补压（`BCdiffLevel != 0`）时，排风间歇的开停不会触发关联间歇控制，避免补压逻辑与关联逻辑冲突。

### 4.6 关联间歇完整运行时序

假设配置：排风变频间歇 opTime=280秒, clTime=20秒，送风变频关联 (CtrMode=1)：

```
时间轴(秒):
0         280       300       580       600
|----排风开----|--排风关--|----排风开----|--排风关--|
|  state=1    | state=2  |  state=1    | state=2  |

送风变频:
|  fc_runFreq[1] |fc_runFreq[0]|  fc_runFreq[1] |fc_runFreq[0]|
|  (较高频率)     | (较低频率)   |  (较高频率)     | (较低频率)   |

微正压维持:
|  排风运行中     | 排风停止     |  排风运行中     | 排风停止     |
|  送风较高       | 送风降低     |  送风较高       | 送风降低     |
|  → 正压充足     | → 正压降低   |  → 正压充足     | → 正压降低   |
|                 | 送风补偿     |                 | 送风补偿     |
```

### 4.7 关联间歇逻辑表示例（具体场景）

以下以图片中的实际案例为例，展示关联间歇控制在通风等级5下的具体运行逻辑。

**场景配置**：
- 通风等级：5级
- 送风变频风机组1：配置为 `linkBreak`（间歇关联）模式
- 抽风定速风机组1：配置为 `interval`（间歇）模式
- 间歇参数：opTime=35秒, clTime=20秒
- 送风变频频率：
  - 排风开启时：75%（fc_runFreq[1]）
  - 排风关闭时：43%（fc_runFreq[0]）

**逻辑表**：

| 时间点 | 排风状态 | 送风变频频率 | 事件说明 |
|:---|:---|:---|:---|
| t=0s | ON（开启） | 75% | 通风等级5执行，排风定速风机开启，送风变频以75%运行 |
| t=35s | OFF（关闭） | 43% | 排风间歇opTime到，排风关闭，送风变频降频至43%维持微正压 |
| t=55s | ON（开启） | 75% | 排风间歇clTime到，排风开启，送风变频恢复75% |
| t=90s | OFF（关闭） | 43% | 排风间歇opTime到，排风关闭，送风变频降频至43% |
| t=110s | ON（开启） | 75% | 排风间歇clTime到，排风开启，送风变频恢复75% |
| ... | ... | ... | 循环往复 |

**状态转换表**：

| 排风间歇状态 | 送风关联动作 | 物理效果 |
|:---|:---|:---|
| state=1（开启周期） | 送风变频 = fc_runFreq[1] = 75% | 排风抽气，送风补气，正压充足 |
| state=2（关闭周期） | 送风变频 = fc_runFreq[0] = 43% | 排风停止，送风降低频率维持微正压 |

**关键逻辑点**：
1. 排风定速风机组1执行间歇操作时，每35秒开启、20秒关闭，形成55秒周期
2. 送风变频风机组1根据排风状态实时调整频率：排风开时75%，排风关时43%
3. 这种"先送后抽、先减抽后减送"的节奏，确保猪舍微正压环境稳定
4. 送风变频的频率切换由关联间歇控制器 `ventilation_fan_relate_controller` 自动完成，无需人工干预

## 5、三种模式的交互与优先级

### 5.1 模式选择流程

当通风级别执行时，风机的 workMode 决定了走哪条控制路径：

```
通风级别执行
    │
    ├─ workMode == trensRun (1)  → 全时开启，直接输出
    ├─ workMode == Fanclose (2)  → 关闭，直接输出
    ├─ workMode == interval (0)  → 间歇控制
    │   └─ 初始化 s_interval_info，设置 state=1
    │      由 1s 定时器驱动 tick 函数
    │
    ├─ workMode == takeTurns (3) → 全速轮替
    │   └─ 由 ventilation_rotation_controller 管理
    │      在主循环中周期调用
    │
    ├─ workMode == takeBreak (4) → 间歇轮替
    │   └─ 由 ventilation_rotation_controller 管理
    │      在主循环中周期调用
    │
    └─ workMode == linkBreak (5) → 间歇关联
        └─ 由排风间歇 tick 触发
           ventilation_fan_relate_controller 执行
```

### 5.2 间歇与轮替的互斥关系

在 `ventilation_fan_executor_run_pwm_level()` 中：

```c
if ((workMode == takeTurns) || (workMode == takeBreak)) {
    return;  // 轮替模式下，PWM风机由轮替控制器管理，此处直接返回
}
if (workMode <= trensRun) {
    // 非间歇模式，清除间歇状态
    intervalInfo->state = 0;
}
```

**含义**：间歇和轮替是互斥的。如果风机配置了轮替模式，间歇控制器不会干预它；反之亦然。

### 5.3 关联间歇与排风间歇的耦合

关联间歇不是一个独立的定时器驱动模式，而是**嵌套在排风间歇 tick 函数中的联动逻辑**：

```
排风间歇 tick (每1秒)
    │
    ├─ state=1 (开启周期)
    │   └─ tick >= opTime → 关闭排风机
    │       └─ 触发 ventilation_fan_relate_controller_apply(STATE_OFF)
    │           └─ 根据 CtrMode 联动送风侧
    │
    └─ state=2 (关闭周期)
        └─ tick >= clTime → 开启排风机
            └─ 触发 ventilation_fan_relate_controller_apply(STATE_ON)
                └─ 根据 CtrMode 联动送风侧
```

### 5.4 送风侧与排风侧的独立性

| 维度 | 送风侧 | 排风侧 |
|:---|:---|:---|
| 间歇状态机 | `s_interval_info[0]` | `s_interval_info[1]` |
| 逻辑表来源 | `logicEnControl[]` | `logicAirPumping[]` |
| 间歇 tick 函数 | `tick_supply_interval()` | `tick_exhaust_interval()` |
| 轮替控制器 | `run_supply()` | `run_exhaust()` |
| 关联触发 | 被动接收 | 主动触发 |

两侧可以独立配置不同的工作模式。例如：送风侧配置 `trensRun`（全时开启），排风侧配置 `interval`（间歇），此时只有排风侧会进入间歇状态机。
