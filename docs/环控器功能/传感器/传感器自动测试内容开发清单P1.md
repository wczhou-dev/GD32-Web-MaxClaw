## 一、测试概述

### 1.0 评审结论

本文档的测试方向基本合理，已经覆盖传感器模块 P1 阶段最核心的功能面：正常抄读、通信异常、数值不变异常、历史数据回退、配置热更新、平均值计算和综合并发场景。

但从自动化闭环角度看，原清单仍偏向人工或物理环境验证，例如拔线、温箱、标准仪器、调试串口监听等。P1 阶段建议补齐“传感器模拟器 + 场景期望表 + 自动断言”的工程化描述，让测试系统能够预先控制模拟器返回值，再读取环控器处理结果并自动判断通过/失败。

配套文档分工如下：

| 文档 | 定位 | 说明 |
| :--- | :--- | :--- |
| 本文档 | 测什么 | 定义 P1 自动化测试项、优先级和验收范围 |
| `传感器测试场景与期望表.md` | 给什么输入、期望什么输出 | 定义每个场景的模拟器输入、环控器期望值、断言规则 |
| `传感器模拟器实现方案.md` | 怎么实现模拟器 | 定义 PC 端 RS485 从站模拟器的协议、接口和代码落点 |

### 1.1 测试范围

本文档涵盖传感器模块的自动化测试内容，包括：

- 传感器异常自动过滤温湿度
- 传感器异常使用历史数据
- 传感器正常抄读
- 传感器配置改变热更新（无需重启）
- 传感器计算平均值


### 1.4 自动化闭环原则

每个自动化用例都应形成如下闭环：

1. 测试系统加载场景，明确本次模拟器应返回的传感器值或故障状态。
2. 测试系统调用传感器模拟器接口，写入影子寄存器或注入故障。
3. 环控器通过 RS485 主动发送抄读指令。
4. 模拟器按场景返回 Modbus RTU 数据，记录请求帧、响应帧、解析值和时间戳。
5. 测试系统读取环控器侧结果，例如 ATE TCP、调试寄存器、串口日志或导出的状态接口。
6. 断言引擎对比“模拟器预置值/故障”与“环控器处理结果”，生成测试报告。

因此，P1 阶段不能只维护一份测试项清单，还必须同步维护可执行的场景/期望表，避免测试时临时人工判断。

## 二、传感器正常抄读测试

### 2.0 场区适配（前置步骤）

不同场区的传感器 Modbus RTU 从站地址存在差异（见 §十三），测试系统在读取传感器数据前，**必须先读取环控器的场区类型，再加载对应的传感器地址配置**。

**读取流程**：

```
┌─────────────────────────────────────────────────────────┐
│ 1. Modbus TCP 连接环控器 (端口 502)                      │
│ 2. 读取场区类型寄存器 0x0019 (readHoldingRegisters)       │
│ 3. 根据返回值加载对应场区的传感器地址配置表                  │
│    ├─ 0 → 未配置，提示用户先通过 HMI 设置场区              │
│    ├─ 1 → 配置类型 A（PDF 第1~4页）                      │
│    ├─ 2 → 配置类型 B（PDF 第6页）                        │
│    └─ 3 → 配置类型 C（PDF 第7页起）                      │
│ 4. 使用配置表中的地址进行后续传感器数据读取                  │
└─────────────────────────────────────────────────────────┘
```

**场区类型编码**（数据来源：《传感器地址汇总.pdf》）：

| 编码 | 配置类型 | TH7 | TH14/TH15/TH16 | CO2-2 | CO2-8 | 高位H1/H2 | 室外TH | 水帘1/水帘2 |
| :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| 0 | 未配置 | - | - | - | - | - | - | - |
| 1 | A | 0x50 | 0x51/0x33/0x34 | 0x0A | 0x52 | 0x21/0x22 | 0x18 | 0x19/0x20 |
| 2 | B | 0x19 | 0x20/0x21/0x22 | 0x10 | 0x18 | 0x33/0x34 | 0x04 | 0x13/0x12 |
| 3 | C | 0x19 | 0x20/0x21/0x22 | 0x10 | 0x18 | 0x33/0x34 | 0x24 | 0x13/0x12 |

> 场区类型寄存器（`0x0019`）需要环控器固件新增支持，详见《环控器主板需添加内容.md》§一。
> 配置类型 B 与 C 的传感器地址完全相同，仅室外 TH 地址不同（0x04 vs 0x24）。
> 其余传感器地址（TH1~TH6、TH8~TH13、CO2-1/3~7、压差、氨气、风速等）在所有场区中一致，不受场区影响。

**后端初始化伪代码**：

```javascript
// 连接后第一步：读取场区类型
const zoneResult = await client.readHoldingRegisters(0x0019, 1);
const fieldZoneType = zoneResult.data[0];

if (fieldZoneType === 0) {
  throw new Error('环控器未配置场区，请先通过 HMI 设置');
}

// 加载对应场区的传感器地址表
const sensorAddrs = FIELD_CONFIG[fieldZoneType].sensorAddresses;

// 后续使用 sensorAddrs 进行传感器数据读取
```

### 2.1 传感器安装确认（前置步骤）

在读取传感器数据前，必须先确认环控器中对应传感器已配置为"已安装"。若传感器未安装，环控器不会对其进行抄读，模拟器即使返回数据也不会被采集。

**读取方式**：通过 Modbus TCP 读取传感器安装状态寄存器（`0x700A ~ 0x700F`），位掩码格式。

**确认流程**：

```
┌──────────────────────────────────────────────────────────┐
│ 1. Modbus TCP readHoldingRegisters(0x700A, 6)            │
│ 2. 解析位掩码，得到各路传感器安装状态                       │
│ 3. 检查本次测试目标传感器是否已安装                        │
│    ├─ 已安装 → 进入数据读取                               │
│    └─ 未安装 → 执行未安装屏蔽断言，记录 skipReason        │
└──────────────────────────────────────────────────────────┘
```

**寄存器地址**：

| 寄存器 | 传感器类型 | 位映射 |
| :---: | :--- | :--- |
| `0x700A` | 室内温度 | bit0~bit15 → 1#~16# |
| `0x700B` | 室内湿度 | bit0~bit15 → 1#~16# |
| `0x700C` | 室内CO2 | bit0~bit7 → 1#~8# |
| `0x700D` | 室内氨气 | bit0~bit3 → 1#~4# |
| `0x700E` | 室内压差 | bit0~bit3 → 1#~4# |
| `0x700F` | 室外传感器 | bit0=室外温湿度，bit1~bit4=水帘1~4 |

**未安装处理策略**：

> 参考《环境控制器自动化测试系统开发方案P0》§14.1：
> 若传感器判定为"未安装"，对应测试项不应跳过，而是进入"未安装屏蔽"断言——
> 继续模拟满足采集条件，验证环控器确实未对该路进行抄读，报告记录 `skipReason = not_installed_masked`。

**后端实现伪代码**：

```javascript
// 一次读取所有传感器安装状态
const result = await client.readHoldingRegisters(0x700A, 6);
const data = result.data;

const sensorInstalled = {
  temp:    Array.from({length: 16}, (_, i) => Boolean((data[0] >> i) & 1)),
  humi:    Array.from({length: 16}, (_, i) => Boolean((data[1] >> i) & 1)),
  co2:     Array.from({length: 8},  (_, i) => Boolean((data[2] >> i) & 1)),
  nh3:     Array.from({length: 4},  (_, i) => Boolean((data[3] >> i) & 1)),
  press:   Array.from({length: 4},  (_, i) => Boolean((data[4] >> i) & 1)),
  outdoor: Array.from({length: 5},  (_, i) => Boolean((data[5] >> i) & 1)),
};

// 在测试用例中使用
for (let i = 0; i < 16; i++) {
  if (!sensorInstalled.temp[i]) {
    report.addSkip(`temp[${i}]`, 'not_installed_masked');
    continue;
  }
  // 正常执行温度抄读测试...
}
```

### 2.2 数据读取方式

环控器传感器数据通过 **Modbus TCP** 读取，寄存器区为 `BLOCK_ENV`（地址 `0x1001 ~ 0x1048`）。

**协议参数**：
- 传输层：TCP，默认端口 502
- 功能码：`0x03`（读保持寄存器）
- 起始地址：`0x1001`，可一次批量读取最多 72 个寄存器

**寄存器地址速查**：

| 传感器类型 | 起始地址 | 数量 | 地址规则 | 换算 |
| :--- | :---: | :---: | :--- | :--- |
| 室内温度 | `0x1001` | 16路 | `0x1001 + i×2` (i=0..15) | int16, val/10 → ℃ |
| 室内湿度 | `0x1002` | 16路 | `0x1002 + i×2` (i=0..15) | uint16, val/10 → %RH |
| 室内CO2 | `0x1021` | 8路 | `0x1021 + i` (i=0..7) | uint16, 原值 → ppm |
| 室内氨气 | `0x1029` | 4路 | `0x1029 + i` (i=0..3) | uint16, 原值 → ppm |
| 室内风速 | `0x102D` | 12路 | `0x102D + i` (i=0..11) | uint16, val/10 → m/s |
| 舍外温度 | `0x1039` | 1路 | - | int16, val/10 → ℃ |
| 舍外湿度 | `0x103A` | 1路 | - | uint16, val/10 → %RH |
| 室内压差 | `0x1042` | 4路 | `0x1042 + i` (i=0..3) | int16, val/10 → Pa |

> 完整寄存器定义参见 `shared/constants.js`（BLOCK_ENV）和 `docs/Modbus-TCP后端通信开发规范.md`（§7.1 传感器数据帧）。

### 2.3 温湿度传感器抄读

#### 测试项：T-READ-001 室内温度传感器抄读

```
测试目的：验证室内温度传感器数据采集正确性
前置条件：
  1. 系统正常运行，16 路室内温度传感器已部署
  2. 传感器模拟器已启动，影子寄存器中预设各路温度值
测试步骤：
  1. 通过模拟器为 16 路温度传感器预设不同温度值（如 20.0℃~35.0℃）
  2. 等待环控器完成一轮轮询采集（约 10~15 秒）

  4. 按 val/10 换算后逐路与模拟器预设值对比
  5. 验证每路数据是否完全一致
预期结果：
  1. 环控器读取的 16 路温度值与模拟器预设值完全一致

  3. 所有传感器在线位正常
自动化实现：
  1. 模拟器通过 setSensorValue 设置各路温度值
  2. 等待采集周期完成
  3. readHoldingRegisters(0x1001, 32)，取 temp[i] = val[2*i] / 10
  4. 断言逐路对比：环控器值 == 模拟器预设值
```

## 三、传感器异常自动过滤温湿度测试

### 3.1 通信失败异常过滤

#### 测试项：T-ABNF-001 传感器离线检测

```
测试目的：验证传感器通信失败达到阈值时自动过滤该传感器数据
前置条件：
  1. 系统正常运行，所有传感器在线
  2. 温度传感器正常采集（如第1路温度值25.0℃）
测试步骤：
  1. 模拟器对第1路温度传感器注入不响应故障（injectTimeout）
```

## 十、模拟器实现配套文档与接口要求

### 10.1 模拟器在 P1 自动化中的定位

传感器模拟器不是单独调试工具，而是自动化测试系统的受控输入源。它需要被 `TestManager` 或测试编排层直接调用，按测试场景动态改变传感器返回值、通信状态和异常行为。

建议正式实现放在当前工程后端：

| 模块 | 建议文件 | 责任 |
| :--- | :--- | :--- |
| 传感器模拟器 | `backend/ate/SensorSimulator.js` | 打开 USB-RS485 串口，解析环控器 Modbus RTU 请求，按影子寄存器回复 |
| 场景目录 | `backend/ate/TestScenarioCatalog.js` | 加载传感器场景和期望表，统一管理 scenarioId |
| 控制器读取 | `backend/ate/ControllerStateReader.js` | 从 ATE TCP、调试串口或寄存器读取环控器实际处理结果 |
| 断言引擎 | `backend/ate/AssertEngine.js` | 比较 expected 与 actual，输出通过/失败原因 |
| 测试编排 | `backend/ate/TestManager.js` | 串联 applyScenario、等待采集、读取结果、断言、报告 |

### 10.2 必须具备的模拟器能力

| 能力 | 说明 | 对应用例 |
| :--- | :--- | :--- |
| 多从站地址 | 同一 USB-RS485 口模拟多路温湿度、压差、CO2、NH3 等从站 | 正常抄读、多路异常 |
| 影子寄存器表 | 保存每个从站、寄存器地址、原始值、工程值、单位和缩放系数 | 所有闭环测试 |
| 动态改值 | 测试过程中实时调整温度、湿度、压差、气体浓度 | 环控逻辑、平均值、热更新 |
| 不响应/超时 | 指定从站或指定寄存器不回复 | ErRead、离线恢复 |
| 固定值不变 | 连续返回相同值 | ErMax 数值不变异常 |
| 异常跳变 | 返回明显偏离中位数或越界值 | 偏差剔除、异常过滤 |
| CRC 错误 | 故意返回错误 CRC | 通信异常鲁棒性 |
| 延迟响应 | 模拟总线慢响应 | 超时边界、轮询稳定性 |
| 交易日志 | 记录请求帧、响应帧、解析值、场景 ID、时间戳 | 问题定位、报告证据 |

### 10.3 模拟器接口契约

模拟器至少提供以下内部接口，供 `TestManager` 调用：

```js
await sensorSimulator.start({
  port: 'COMx',
  baudRate: 9600,
  parity: 'none',
  dataBits: 8,
  stopBits: 1
})

await sensorSimulator.stop()

await sensorSimulator.applyScenario({
  scenarioId: 'SEN-TEMP-001',
  sensors: [
    { key: 'temp_1', slaveAddr: 1, value: 20.0, unit: 'C', scale: 10 },
    { key: 'humi_1', slaveAddr: 1, value: 65.0, unit: '%RH', scale: 10 }
  ],
  faults: []
})

await sensorSimulator.setSensorValue('temp_1', 25.5)
await sensorSimulator.setRegister({ slaveAddr: 1, registerAddr: 0x0000, rawValue: 255 })
await sensorSimulator.injectTimeout({ key: 'temp_1' })
await sensorSimulator.injectFixedValue({ key: 'temp_1', value: 25.0, repeat: 100 })
await sensorSimulator.injectOutlier({ key: 'temp_1', value: 80.0 })
await sensorSimulator.injectCrcError({ key: 'temp_1', count: 3 })
await sensorSimulator.clearFault('temp_1')

const shadow = sensorSimulator.getShadowRegisters()
const log = sensorSimulator.getTransactionLog({ scenarioId: 'SEN-TEMP-001' })
await sensorSimulator.reset()
```

### 10.4 影子寄存器表要求

影子寄存器表是测试系统知道“模拟器返回了什么”的核心数据源。每个传感器点位至少应包含：

| 字段 | 示例 | 说明 |
| :--- | :--- | :--- |
| `key` | `temp_1` | 测试系统内部唯一名称 |
| `sensorType` | `temperature` | 传感器类型 |
| `slaveAddr` | `1` | Modbus RTU 从站地址 |
| `registerAddr` | `0x0000` | 数据寄存器地址 |
| `rawValue` | `200` | 实际回复的寄存器原始值 |
| `value` | `20.0` | 工程值 |
| `scale` | `10` | 工程值到寄存器值的缩放倍数 |
| `unit` | `C` | 单位 |
| `status` | `normal` | `normal/timeout/fixed/outlier/crcError` |
| `updatedAt` | `2026-06-11T10:00:00+08:00` | 最近更新时间 |

```python
# CRC16 计算
def calculate_crc16(data):
    crc = 0xFFFF
    for byte in data:
        crc ^= byte
        for _ in range(8):
            if crc & 0x0001:
                crc = (crc >> 1) ^ 0xA001
            else:
                crc >>= 1
    return crc

# 验证请求报文
request = bytes([0x01, 0x03, 0x00, 0x00, 0x00, 0x02])
expected_crc = calculate_crc16(request)
print(f"请求 CRC: 0x{expected_crc:04X}")  # 应为 0xC40B

# 验证响应报文
response_data = bytes([0x01, 0x03, 0x04, 0x00, 0xFA, 0x00, 0xFA])
expected_crc = calculate_crc16(response_data)
print(f"响应 CRC: 0x{expected_crc:04X}")
```

### 13.7 模拟器场区切换接口

```javascript
// SensorSimulator.js - 场区切换接口

class SensorSimulator {
  /**
   * 加载场区配置
   * @param {string} fieldType - 场区类型: 'typeA' | 'typeB' | 'typeC'
   */
  async loadFieldConfig(fieldType) {
    const configPath = `./config/field_${fieldType}.yaml`;
    const config = await this.readYaml(configPath);
    
    // 更新影子寄存器表
    this.shadowRegisters = {};
    
    // 初始化温湿度传感器
    for (const addr of config.temperature_humidity.indoor_temp.addresses) {
      this.shadowRegisters[addr] = {
        type: 'temperature_humidity',
        address: addr,
        registers: { 0x0000: 250, 0x0001: 500 },  // 温度25.0℃, 湿度50.0%
        scale: 10
      };
    }
    
    // 初始化 CO2 传感器
    for (const addr of config.co2.addresses) {
      this.shadowRegisters[addr] = {
        type: 'co2',
        address: addr,
        registers: { 0x0000: 400 },  // CO2 400ppm
        scale: 1
      };
    }
    
    // ... 其他传感器初始化
    
    console.log(`[模拟器] 已加载场区配置: ${fieldType}`);
    console.log(`[模拟器] 影子寄存器数量: ${Object.keys(this.shadowRegisters).length}`);
  }
  
  /**
   * 获取当前场区配置
   */
  getFieldConfig() {
    return this.currentFieldConfig;
  }
  
  /**
   * 切换场区
   */
  async switchField(fieldType) {
    await this.stop();
    await this.loadFieldConfig(fieldType);
    await this.start();
  }
}

// 使用示例
const simulator = new SensorSimulator();

// 测试场区一
await simulator.loadFieldConfig('typeA');
await simulator.start();

// 切换到配置 C
await simulator.switchField('typeC');
```

## 十四、建议首批落地顺序

为了降低 P1 实现风险，建议按以下顺序开发和验证：

1. 先实现 `SensorSimulator` 的正常 Modbus RTU 回复能力，只覆盖 1 路温度和 1 路湿度。
2. 接入 `TestManager.applyScenario`，让测试系统能够把场景值写入模拟器。
3. 实现交易日志，确保测试失败时能看到环控器请求和模拟器响应。
4. 跑通 `正常抄读 -> 读取环控器结果 -> 自动断言` 的最小闭环。
5. 再扩展通信超时、固定值不变、异常跳变、CRC 错误等故障注入能力。
6. 最后接入平均值、历史回退、热更新和环控逻辑联动场景。

首批建议落地场景：

| 优先级 | 场景 | 原因 |
| :---: | :--- | :--- |
| P0 | 单路温度正常抄读 | 验证模拟器、环控器、断言链路是否打通 |
| P0 | 单路温度通信失败 ErRead | 验证最关键的异常闭环 |
| P0 | 多路温度平均值 | 验证影子寄存器与期望表能支撑聚合计算 |
| P1 | 固定值不变 ErMax | 验证长周期异常注入 |
| P1 | 配置热更新后改地址/部署位 | 验证场景切换与轮询队列重建 |
| P1 | 温度阶跃触发环控逻辑 | 验证传感器测试向环控系统测试扩展 |
