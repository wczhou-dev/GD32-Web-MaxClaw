## 二、项目总体说明

### 2.1 产品定位

本项目是**环控器自动化测试系统**，用于对 GD32 系列环控器（环境控制器）进行硬件在环（HIL）自动化测试。系统具备以下核心能力：

| 能力 | 说明 |
| :--- | :--- |
| **硬件在环测试（HIL）** | 通过 Modbus TCP 协议与真实环控器硬件或模拟环控器通信 |
| **传感器模拟** | 虚拟传感器数据注入，支持多种传感器类型（温湿度、CO2、NH3、压差、风速等） |
| **OTA 固件升级管理** | 固件包上传、校验、升级流程管理 |
| **自动化测试框架** | 支持测试用例编写、执行、报告生成 |
| **实时数据监测** | 后台线程实时采集 Modbus 数据，前端仪表板展示 |
| **ATE TCP+JSON 协议** | 与固件端 ATE 服务直接通信，支持高可靠测试控制 |
| **MSH 调试串口** | 通过串口发送 shell 命令，读取固件内部状态 |

### 2.2 关键业务场景

1. **硬件自检**：验证环控器 9 项基础硬件（SPI Flash、EEPROM、RTC、RS485×2、CAN、ADC/AO、22 路继电器、RS485 热插拔）
2. **传感器读数验证**：通过 SensorSimulator 注入模拟传感器值，验证控制器读数精度和异常过滤逻辑
3. **异常过滤测试**：验证 ErRead（通信超时）、ErMax（值不变）、偏差剔除等异常处理机制
4. **历史数据回退**：验证跨小时历史缓冲区写入、掉电重启后数据恢复
5. **配置热更新**：验证传感器使能/禁用、端口切换、阈值修改、补偿值修改等热更新能力
6. **OTA 固件升级**：固件包上传到本地 HTTP 服务器，触发控制器通过 HTTP 拉取升级
7. **长期稳定性监测**：实时采集传感器数据并在前端仪表板展示

## 五、共享常量模块详解

> **文件**：`shared/constants.js`（1015 行）

这是整个系统的**单一事实来源（Single Source of Truth）**，为前后端提供统一的寄存器地址、状态码、错误码、协议帧格式等常量，消除魔数硬编码。

### 5.1 ATE 测试控制与状态寄存器区 (BLOCK_TEST_STATUS)

地址范围：`0x8000 - 0x802F`，共 48 个寄存器。

| 寄存器地址 | 常量名 | 读写 | 类型 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 0x8000 | `START` | R/W | uint16 | 控制命令：0=idle, 1=start, 2=stop, 3=reset |
| 0x8001 | `VENTILATION_LEVEL` | RO | uint16 | 当前通风等级 |
| 0x8002 | `CURRENT_ITEM` | RO | uint16 | 当前正在测试的项目 ID |
| 0x8003 | `PROGRESS` | RO | uint16 | 测试进度百分比（0-100） |
| 0x8004 | `OVERALL_STATUS` | RO | uint16 | 整体测试状态码 |
| 0x8005 | `SESSION_ID_HIGH` | RO | uint16 | 会话 ID 高 16 位 |
| 0x8006 | `SESSION_ID_LOW` | RO | uint16 | 会话 ID 低 16 位 |
| 0x8007 | `ERROR_ITEM_ID` | RO | uint16 | 失败项 ID |
| 0x8008 | `TEST_MASK` | R/W | uint16 | 测试项掩码（0x01FF = 全部 9 项） |
| 0x8010-0x8018 | `SINGLE_RESULT_BASE` | RO | uint16[9] | 单项自检结果码 |
| 0x8020-0x8028 | `ERROR_CODE_BASE` | RO | uint16[9] | 故障错误码 |
| 0x802C | `DIAG_CHANNEL` | RO | uint16 | 诊断通道号 |
| 0x802D | `DIAG_EXPECTED` | RO | uint16 | 诊断期望值（mV） |
| 0x802E | `DIAG_ACTUAL` | RO | uint16 | 诊断实际值（mV） |

### 5.2 ATE 测试参数配置区 (BLOCK_TEST_CONFIG)

地址范围：`0x8030 - 0x803F`。

| 寄存器地址 | 常量名 | 读写 | 说明 |
| :--- | :--- | :--- | :--- |
| 0x8030-0x8033 | `AO_TARGET_1~4` | R/W | AO 测试目标电压（mV），默认 2500 |
| 0x8034 | `RELAY_DI_MAP` | R/W | 继电器-DI 闭环映射 |
| 0x8035 | `RS485_SLAVE_ID` | R/W | RS485 从站 ID 配置 |
| 0x8038 | `DEVICE_MODEL` | R/W | 设备型号：0=9200，1=9250/9300 |
| 0x8039 | `VENT_MODE` | R/W | 通风逻辑：0=负压通风，1=微正压通风 |

### 5.3 环境数据寄存器区 (BLOCK_ENV)

地址范围：`0x1001 - 0x1050`，存放所有传感器采集数据。

| 地址 | 常量名 | 类型 | 换算 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 0x1001 | `TEMP_1` | int16 | val/10 | 1# 舍内温度（℃） |
| 0x1002 | `HUMI_1` | uint16 | val/10 | 1# 舍内湿度（%RH） |
| 0x1003-0x1020 | TEMP_2~HUMI_16 | int16/uint16 | val/10 | 2#-16# 温湿度 |
| 0x1021 | `CO2_1` | uint16 | 原值 | 1# CO2 浓度（ppm） |
| 0x1021-0x1028 | CO2_1~CO2_8 | uint16 | 原值 | 1#-8# CO2 |
| 0x1029 | `NH3_1` | uint16 | 原值 | 1# 氨气浓度（ppm） |
| 0x1029-0x102C | NH3_1~NH3_4 | uint16 | 原值 | 1#-4# 氨气 |
| 0x102D | `WIND_1` | uint16 | val/10 | 1# 风速（m/s） |
| 0x102D-0x1038 | WIND_1~WIND_12 | uint16 | val/10 | 1#-12# 风速 |
| 0x1039 | `OUTDOOR_TEMP` | int16 | val/10 | 舍外温度 |
| 0x103A | `OUTDOOR_HUMI` | uint16 | val/10 | 舍外湿度 |
| 0x103B | `ACTUAL_TEMP` | int16 | val/10 | 计算实际温度（加权平均） |
| 0x103C | `ACTUAL_HUMI` | uint16 | val/10 | 计算实际湿度 |
| 0x1042-0x1045 | PRESS_1~PRESS_4 | int16 | val/10 | 1#-4# 压差（Pa） |

| 地址 | 常量名 | 说明 |

地址范围：`0x700A - 0x700F`，存放传感器安装位掩码。

| 地址 | 常量名 | 说明 |
| :--- | :--- | :--- |

### 6.2 设备连接池 (DevicePool)

> **文件**：`backend/DevicePool.js`（378 行）

#### 6.2.1 类定义

```javascript
class DevicePool {
  // 内部状态
  devices: Map<string, {client: ModbusRTU, info: Object, status: string}>
  eventHandlers: Map<string, Function[]>
  _exclusiveQueues: Map<string, Promise>      // 每设备串行执行队列
  _cooldownTimestamps: Map<string, number>    // 重连冷却时间戳
}
```

#### 6.2.2 设备键格式

设备唯一标识为 `${ip}:${port}:${unitId}`，例如 `192.168.110.125:502:1`。

#### 6.2.3 核心方法详解

**`addDevice(config)`**
- 验证 config 包含 ip/port/unitId
- 创建 `ModbusRTU` 客户端实例
- 注册到 `devices` Map
- 设备初始状态为 `INIT`

**`async connect(key)`**
- 检查冷却期（12 秒内不允许重连）
- 设置状态为 `CONNECTING`
- 调用 `client.connectTCP(ip, {port})`
- 启用 TCP keepalive（10 秒）
- 设置 Modbus 超时（5 秒）
- 成功后状态为 `CONNECTED`，失败为 `DISCONNECTED`

**`async runExclusive(key, task)`**
- 保证同一设备同一时刻只有一个 Modbus 操作在执行
- 通过 Promise 链实现串行化
- 所有 read/write 操作都通过此方法排队

**`async readHoldingRegisters(key, address, length)`**
- 调用 `runExclusive` 串行化
- 3 次超时硬重置逻辑：连续 3 次超时后强制断开重连
- 返回寄存器数组

**`async writeRegister(key, address, value)` / `writeRegisters(key, address, values)`**
- 同样的 3 次超时硬重置逻辑
- 写入单个或多个保持寄存器

#### 6.2.4 重连冷却机制

- 断开连接时记录 `_cooldownTimestamps`
- 冷却期 12 秒（`CONFIG_DEFAULTS.ATE_RECONNECT_COOLDOWN_MS`）
- 冷却期内调用 `connect()` 会跳过
- `isInCooldown(key)` / `getCooldownRemaining(key)` 用于查询

#### 6.2.5 事件系统

| 事件 | 参数 | 说明 |
| :--- | :--- | :--- |
| `statusChange` | `(key, status)` | 设备状态变更：INIT/CONNECTING/CONNECTED/DISCONNECTED |

#### 6.2.6 设备连接状态机

```mermaid
stateDiagram-v2
    [*] --> INIT : addDevice()

    INIT --> CONNECTING : connect()
    CONNECTING --> CONNECTED : TCP 握手成功
    CONNECTING --> DISCONNECTED : TCP 握手失败

    CONNECTED --> DISCONNECTED : 超时/错误
    CONNECTED --> DISCONNECTED : disconnect()

    DISCONNECTED --> CONNECTING : connect()\n(冷却期过后)
    DISCONNECTED --> DISCONNECTED : connect()\n(冷却期内, 跳过)

    CONNECTED --> CONNECTED : readHoldingRegisters()\nwriteRegister()
    CONNECTED --> DISCONNECTED : 3次超时硬重置

    state CONNECTED {
        [*] --> Idle
        Idle --> Reading : readHoldingRegisters
        Reading --> Idle : 返回数据
        Idle --> Writing : writeRegister
        Writing --> Idle : 写入完成
        Idle --> Exclusive : runExclusive
        Exclusive --> Idle : 任务完成
    }
```

#### 6.2.7 Modbus 读写时序图

```mermaid
sequenceDiagram
    participant Caller as 调用方
    participant DP as DevicePool
    participant Queue as _exclusiveQueues
    participant Client as ModbusRTU
```

### 6.3 数据采集引擎 (PollingEngine)

> **文件**：`backend/PollingEngine.js`（370 行）

#### 6.3.1 类定义

```javascript
class PollingEngine {
  pool: DevicePool                    // 设备连接池引用
  parser: DataParser                  // 数据解析器
  config: { intervalMs: 2000 }        // 轮询间隔
  isRunning: boolean                  // 运行状态
  timerId: number                     // setInterval 句柄
  heartbeatCounter: number            // 心跳计数器
  writeLock: boolean                  // 写操作互斥锁
  deviceStatus: Map<key, {lastHeartbeat, lastSensorPoll}>
  _devicesUnderTest: Set<string>      // 正在 ATE 测试中的设备
  onData: Function                    // 数据回调
  onStatusChange: Function            // 状态回调
}
```

#### 6.3.2 轮询策略

| 操作 | 间隔 | 说明 |
| :--- | :--- | :--- |
| 心跳写入 | 5 秒 | 写寄存器 0x0000 保持连接 |
| 传感器数据采集 | 30 秒 | 读取 BLOCK_ENV_1 + BLOCK_ENV_2 + BLOCK_HW |
| 轮询主循环 | 2 秒 | setInterval 主循环，内部按计数器分派 |

#### 6.3.3 核心方法

**`start()`** — 启动轮询，带重叠检测（防止上一轮未完成时触发下一轮）

**`async pollDevice(key)`**
- 心跳：每 5 秒写 0x0000
- 传感器：每 30 秒读取三个数据块（BLOCK_ENV_1: 0x1001-0x1028, BLOCK_ENV_2: 0x1029-0x1048, BLOCK_HW: 0x4001-0x400B）
- 解析：`parser.parseSensorData({env, hw})`
- 回调：`onData(deviceKey, sensorData)`

**`markDeviceUnderTest(deviceKey)` / `unmarkDeviceUnderTest(deviceKey)`**
- ATE 测试期间暂停常规轮询，避免干扰测试
- 测试结束后恢复

**`async controlRelay(key, relayIndex, value)`**
- 读取当前继电器状态（0x5001-0x5002）
- 使用 `parser.generateRelayControl()` 位操作设置/清除目标位
- 写回寄存器
- 验证回读（readback）
- 使用 `writeLock` 互斥锁防止并发写入

**`async triggerOTA(key, version)`**
- 写版本号到 0x0101
- 写触发命令到 0x0100

#### 7.4.3 流式解析器 (feed)

- 处理 TCP 粘包/拆包：内部维护 `_buffer` 累积器
- 扫描 Magic 0x55AA 定位帧边界
- 解析帧头提取 CmdType 和 Length
- 验证 Length 不超过 MAX_JSON_SIZE
- 解析 JSON Payload
- 返回解析出的帧数组

#### 7.4.4 AteFrameCodec 帧解析活动图

```mermaid
flowchart TD
    Start([feed data]) --> Append[追加到 _buffer]
    Append --> Scan{扫描 0x55AA<br/>_findMagic}

    Scan -->|未找到| WaitMore[等待更多数据]
    WaitMore --> Return1([返回空数组])

    Scan -->|找到 Magic 位置| Trim[丢弃 Magic 前的垃圾字节]
    Trim --> Enough{buffer.length >= 6?<br/>(Magic+CmdType+Length)}

    Enough -->|否| WaitMore2[等待更多数据]
    WaitMore2 --> Return2([返回空数组])

    Enough -->|是| ParseHeader[解析帧头:<br/>CmdType = buffer.readUInt16BE(2)<br/>Length = buffer.readUInt16BE(4)]
    ParseHeader --> CheckLen{Length > MAX_JSON_SIZE?<br/>(16384)}

    CheckLen -->|是| DropFrame[丢弃损坏帧<br/>stats.errored++]
    DropFrame --> Advance1[跳过 Magic, 继续扫描]
    Advance1 --> Scan

    CheckLen -->|否| EnoughData{buffer.length >= 6+Length?}

    EnoughData -->|否| WaitMore3[等待更多数据]
    WaitMore3 --> Return3([返回已解析帧])

    EnoughData -->|是| ExtractFrame[提取帧:<br/>frameBuffer = buffer.slice(0, 6+Length)<br/>buffer = buffer.slice(6+Length)]
    ExtractFrame --> ParseJSON{JSON.parse(payload)}

    ParseJSON -->|成功| PushFrame[frames.push Frame<br/>{cmdType, payload}<br/>stats.decoded++]
    ParseJSON -->|失败| DropBad[丢弃帧<br/>stats.errored++]

    PushFrame --> MoreData{buffer 还有数据?}
    DropBad --> MoreData

    MoreData -->|是| Scan
    MoreData -->|否| ReturnFrames([返回 frames[]])
```

#### 7.9.2 各场景类型执行详情

**前置检查 (_executePreCheck)**
- 读取场区类型寄存器（0x0019）
- 读取传感器安装状态（0x700A-0x700F）
- 读取环境数据块验证基本通信

**正常抄读 (_executeNormalRead)**
- 通过 SensorSimulator 设置传感器值
- 等待控制器轮询周期（温度 15 秒、压差 35 秒、CO2 90 秒）
- 通过 ControllerStateReader 读回数据
- AssertEngine 逐传感器比对精度

**异常过滤 (_executeAbnormalFilter)**
- **ErRead**：注入超时故障 → 等待 30 次失败累积（约 180 秒）→ 验证 ErRead 标志
- **ErMax**：注入固定值故障 → 等待 100 次重复（约 400 秒）→ 验证 ErMax 标志
- **偏差剔除**：16 路中注入 1 路异常值 → 等待 200 秒 + 60 秒预稳定 → 验证 Actual 值排除异常路

**历史回退 (_executeHistoryFallback)**
- 冻结阶段：同步时间到 HH:59 → 等待跨小时 → 验证历史缓冲区
- 启动回退阶段：注入全传感器超时 → 重启控制器 → 重新同步时间 → 等待 ErRead → 验证 ActualTemp/ActualHumi 匹配冻结值

**配置热更新 (_executeConfigHotUpdate)**
- 传感器使能/禁用：修改安装位 → 验证数据停止/恢复
- 端口切换：修改端口配置 → 验证数据路由变更
- 阈值修改：写入新阈值 → 回读验证 → 恢复原值
- 补偿修改：写入补偿值 → 验证读数偏移 → 恢复原值

**综合场景 (_executeComposite)**
- 异常恢复：模拟传感器离线 → 恢复 → 验证数据正常
- 多路同时故障：同时注入多路异常 → 验证过滤和恢复

#### 7.9.3 关键工具方法

| 方法 | 说明 |
| :--- | :--- |
| `_waitCollect(ms)` | 等待期间每 5 秒发送心跳（读 0x0000），防止 GD32 15 秒 TCP 超时断连 |
| `_waitCrossHour(expectedHour, maxWaitSec, syncResult)` | 多阶段等待跨小时：快速 HR13 轮询(10s) → 时间推算回退 → 强制验证 |

#### 7.10.1 功能概述

SensorSimulator 在 USB-RS485 串口上扮演 Modbus RTU 从站，响应控制器的传感器读取请求。支持真实串口和 Mock 模式。

#### 7.10.2 内部状态

```javascript
_shadowRegisters: Map<slaveAddr, Map<registerAddr, rawValue>>  // 影子寄存器
_faults: Map<key, { type, persist, params }>                   // 故障注入表

_transactionLog: CircularBuffer                                 // 事务日志
```

#### 7.10.3 传感器状态枚举 (SENSOR_STATUS)

| 状态 | 说明 | 行为 |
| :--- | :--- | :--- |
| `NORMAL` | 正常 | 返回设定值 |
| `TIMEOUT` | 超时 | 不响应（无回复帧） |
| `FIXED` | 固定值 | 始终返回同一值 |
| `OUTLIER` | 离群值 | 返回异常偏大/偏小值 |
| `CRC_ERROR` | CRC 错误 | 返回故意损坏 CRC 的帧 |

#### 7.10.4 故障注入方法

| 方法 | 故障类型 | 参数 |
| :--- | :--- | :--- |
| `injectTimeout({key, persist})` | 无响应 | persist=true 持续 |
| `injectFixedValue({key, value, repeat})` | 固定值 | repeat 次数 |
| `injectOutlier({key, value})` | 离群值 | value 异常值 |
| `injectCrcError({key, count})` | CRC 错误 | count 次数 |
| `clearFault(key)` | 清除故障 | — |
| `clearAllFaults()` | 清除全部 | — |

#### 7.10.5 Modbus RTU 协议支持

| 功能码 | 说明 | 行为 |
| :--- | :--- | :--- |
| FC03 | 读保持寄存器 | 从影子寄存器返回 |
| FC04 | 读输入寄存器 | 共享影子寄存器表 |
| FC06 | 写单个寄存器 | 更新影子寄存器 |
| FC010 (0x10) | 写多个寄存器 | 批量更新 |

- 响应延迟：20ms（Modbus 3.5 字符时间）
- 支持异常响应（非法功能码）
- 故障注入时返回故意损坏的 CRC

#### 7.10.6 Mock 模式

当 `mock=true` 或无串口硬件时：
- `mockReadHoldingRegisters(slaveAddr, startAddr, count)` — 内存读取
- `mockGetSensorValue(key)` — 读取工程值
- 不打开物理串口

#### 7.11.3 特殊操作

**`syncTime({year, month, day, hour, minute, second})`**
1. 写入 HR10-HR15（年/月/日/时/分/秒）
2. 写入 HR16=1 触发同步
3. 轮询 HR17 直到为 0（同步完成）
4. 读回设备时间验证
5. 包含固件断连保护（同步时 GD32 可能断开 TCP）

**`reboot({waitMs, retryIntervalMs})`**
1. 写入 HR18=0x55AA（重启魔数）
2. 断开连接
3. 等待指定时间（默认 10 秒）
4. 重连并验证

**`_ensureConnected()`**
- 最多 5 次重试，间隔 3 秒
- 等待冷却期（13 秒）
- 通过读 HR13 验证连接

## 十一、通信协议详解

### 11.1 Modbus TCP 协议

- **角色**：后端作为 Modbus TCP 客户端，控制器作为从站
- **默认端口**：502 或 1502
- **Unit ID**：1
- **支持功能码**：FC03（读保持寄存器）、FC06（写单寄存器）、FC16（写多寄存器）
- **超时**：5000ms
- **Keepalive**：10000ms
- **重连冷却**：12000ms

### 11.2 ATE TCP+JSON 协议

- **端口**：9001
- **帧格式**：`[0x55AA][CmdType 2B][Length 2B][JSON Payload]`
- **所有字段大端序**
- **最大 JSON 体**：16384 字节
- **ACK 超时**：2000ms
- **心跳间隔**：5000ms
- **重连冷却**：12000ms

#### 请求示例

```json
// CmdType = 0x0001 (DOWNLINK)
{
  "method": "test.start",
  "params": { "testMask": 255 },
  "messageId": 1
}
```

#### ACK 响应示例

```json
// CmdType = 0x0003 (ACK)
{
  "messageId": 1,
  "success": true,
  "data": {}
}
```

### 11.3 Modbus RTU 协议（传感器模拟）

- **物理层**：USB-RS485 串口
- **波特率**：9600
- **数据格式**：8N1
- **SensorSimulator 角色**：Modbus RTU 从站
- **控制器角色**：Modbus RTU 主站
- **支持功能码**：FC03、FC04、FC06、FC10
- **响应延迟**：20ms

### 11.4 WebSocket 协议

- **升级方式**：HTTP 升级（与 Express 共享端口）
- **心跳**：Ping/Pong 30 秒间隔
- **消息格式**：JSON `{type: string, payload: object}`
- **自动重连**：前端 5 秒后重连

## 十三、错误处理与容错机制

### 13.1 双协议回退

所有控制器命令优先使用 ATE TCP+JSON，失败时回退到 Modbus TCP。适用于：`test.enter`、`test.exit`、`test.start`、`test.stop`、`test.reset`、`control.force_io`、继电器控制。

### 13.2 Modbus 超时硬重置

DevicePool 中 3 次超时后强制断开重连，防止连接僵死。

### 13.3 重连冷却机制

- DevicePool：12 秒冷却期，防止频繁重连
- AteTcpClient：12 秒冷却期
- ControllerStateReader._ensureConnected()：等待冷却期 + 最多 5 次重试

### 13.4 Keepalive 心跳

- PollingEngine：每 5 秒写 0x0000（Modbus 心跳）
- AteTcpClient：每 5 秒发送 heartbeat 帧
- SensorTestExecutor._waitCollect()：长等待期间每 5 秒读 0x0000，防止 GD32 15 秒 TCP 超时

### 13.5 强韧读写 (Resilient Read/Write)

SensorTestExecutor 中的 `_resilientReadSensorData()`、`_resilientReadActualTempHumi()`、`_resilientWriteCompensation()` 均包含：
- 自动 `_ensureConnected()` 检查
- 最多 3 次重试
- 单次失败后立即重连

### 13.6 安全释放 (_safeRelease)

TestManager 测试完成后始终发送 `test.exit`（TCP）或 `reset`（Modbus），确保 IO 输出不被锁定。

### 13.7 WebSocket 断线重连

前端 deviceStore 在 WebSocket 断开后 5 秒自动重连。AteTest 和 SensorTest 的独立 WebSocket 也有 3 秒重连机制。

### 13.8 进程异常处理

- `uncaughtException`：记录日志，不退出
- `unhandledRejection`：记录日志，不退出
- `SIGINT`：优雅关闭 SensorSimulator

## 十六、常见开发任务指南

### 16.1 新增一个设备

2. 设置正确的 IP、Modbus TCP 端口、Unit ID
3. 重启后端，或动态调用 `devicePool.addDevice(config)`
4. 前端自动通过 WebSocket 刷新设备列表

### 16.2 新增一个传感器类型

3. 在 `backend/ate/fieldConfigs/fieldTypeA.js` 中添加从站地址配置
4. 在 `backend/ate/ControllerStateReader.js` 中添加读写方法
5. 在 `frontend/src/stores/deviceStore.js` 中添加响应式状态
6. 在 `frontend/src/views/Monitor.vue` 中添加展示卡片

### 16.3 新增一个 P1 传感器测试场景

1. 在 `backend/ate/TestScenarioCatalog.js` 中添加场景定义

## 十七、维护注意事项与已知问题

### 17.1 性能注意事项

1. PollingEngine 每 2 秒主循环，每 30 秒采集数据，避免阻塞主线程
2. DevicePool.runExclusive() 保证每设备串行化，防止 Modbus 事务冲突
3. WebSocket 消息过大会影响前端渲染性能
4. 传感器测试期间暂停常规轮询，避免数据竞争

### 17.2 已知问题

| 问题 | 位置 | 说明 |
| :--- | :--- | :--- |
| `Relay.vue` 孤立文件 | frontend/src/views/Relay.vue | 未被路由引用，无法访问 |
| `/sensor-test` 无菜单入口 | frontend/src/App.vue | 侧边栏无对应菜单项 |
| `axios` 未使用 | frontend/package.json | 已声明但未导入，所有 HTTP 用 fetch |
| OTA 参数不匹配 | OTA.vue → deviceStore.triggerOTA() | 调用时未传 deviceIp，版本号被当作第一个参数 |
| `sendWsMessage` 未定义 | AteTest.vue watcher | 引用了 deviceStore 中未定义的方法，运行时会报错 |

### 17.3 安全注意事项

1. OTA 升级过程中不要中断电源，建议添加进度保存机制
2. 手动继电器控制需确认弹窗（Relay.vue 中的 el-popconfirm）
3. 测试报告生成时注意磁盘空间
4. 多设备并发时注意 Modbus 事务 ID 管理
5. 虚拟传感器模拟时不要与真实设备数据混淆

### 17.4 日志管理

- 日志目录：`backend/logs/`
- 双文件轮转：`latest.log` + `previous.log`
- MCU 串口日志：`logs/firmware_runtime.log`
- 前端终端日志上限：系统 500 条，设备 1000 条
