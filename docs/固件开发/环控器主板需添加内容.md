## 一、场区类型寄存器（P0 优先级）

### 1.1 需求背景

不同场区的传感器 Modbus RTU 从站地址存在差异（详见《传感器自动测试内容开发清单P1》§十三）。测试系统和后端在读取传感器数据前，必须先知道当前环控器所处的场区类型，才能加载对应的传感器地址配置表。

**当前问题**：环控器固件中没有通过 Modbus TCP 暴露"场区类型"变量，后端无法自动识别当前场区。

### 1.2 固件改动要求

#### 1.2.1 新增场区类型寄存器

在 `BLOCK_SYS`（地址 `0x0000 ~ 0x0018`）中新增一个只读寄存器，用于标识当前环控器的场区类型。

**建议寄存器地址**：`0x0019`（紧接 BLOCK_SYS 当前末尾 `0x0018` 之后）

| 寄存器地址 | 变量名 | 类型 | 读写 | 说明 |
| :---: | :--- | :---: | :---: | :--- |
| `0x0019` | `FIELD_ZONE_TYPE` | uint16 | RO | 场区类型编码，见下表 |

**场区类型编码定义**（数据来源：《传感器地址汇总.pdf》）：

| 编码值 | 配置类型 | 关键差异 | PDF页码 |
| :---: | :---: | :--- | :---: |
| 0 | 未配置 | 出厂默认值，后端应提示用户配置 | - |
| 1 | A（标准） | TH7=0x50, TH14=0x51, CO2-2=0x0A, 室外TH=0x18 | 第1~4页 |
| 2 | B | TH7=0x19, TH14=0x20, CO2-2=0x10, 室外TH=0x04 | 第6页 |
| 3 | C | 同 B，室外TH=0x24 | 第7页起 |
| 4~255 | 预留 | 后续新场区扩展 | - |

#### 1.2.2 固件实现建议

```c

    FIELD_ZONE_UNCONFIGURED = 0,  // 未配置

    FIELD_ZONE_TYPE_B       = 2,  // 配置 B：PDF 第6页
    FIELD_ZONE_TYPE_C       = 3,  // 配置 C：PDF 第7页起
    // 后续扩展...
} field_zone_type_t;

// 值: field_zone_type_t 枚举值

// === 初始化 ===

// 在系统初始化时，从 EEPROM 或 Flash 读取已保存的场区类型
// 若未配置（值为 0），可通过 HMI 或 Modbus 写入配置

// === HMI 配置入口 ===

// 建议在 HMI 菜单中增加"场区选择"选项，用户选择后保存到 EEPROM
// 写入后立即生效，无需重启（热更新）
```

#### 1.2.3 Modbus 读取示例

```
请求: 01 03 00 19 00 01 XX XX
      │  │     │     │
      │  │     │     └── 读取 1 个寄存器
      │  │     └── 起始地址 0x0019
      │  └── 功能码 0x03
      └── 从站地址 0x01

响应: 01 03 02 00 01 XX XX
      │  │  │  │     │
      │  │  │  │     └── CRC16
      │  │  │  └── 场区类型 = 1（配置 A）
      │  │  └── 字节数 2
      │  └── 功能码 0x03
      └── 从站地址 0x01
```

### 1.3 后端适配要求

后端在连接环控器后、开始轮询传感器数据前，必须先读取场区类型：

```javascript
// ControllerStateReader.js 或 DevicePool 初始化流程

async function initDevice(ip) {
  // 1. 建立 Modbus TCP 连接
  const client = new ModbusRTU();
  await client.connectTCP(ip, { port: 502 });
  client.setID(1);

  // 2. 读取场区类型（0x0019）
  const result = await client.readHoldingRegisters(0x0019, 1);
  const fieldZoneType = result.data[0]; // 0=未配置, 1=配置A, 2=配置B, 3=配置C

  // 3. 根据场区类型加载传感器地址配置
  const fieldConfig = loadFieldConfig(fieldZoneType);

  // 4. 使用配置初始化传感器轮询
  initSensorPolling(client, fieldConfig);
}

function loadFieldConfig(type) {
  // 对应《传感器地址汇总.pdf》三种配置
  const configs = {
    1: { name: 'typeA', th7: 0x50, temp14: 0x51, temp15: 0x33, temp16: 0x34,
         co2_2: 0x0A, co2_8: 0x52, h1: 0x21, h2: 0x22, outdoor: 0x18, wc1: 0x19, wc2: 0x20 },
    2: { name: 'typeB', th7: 0x19, temp14: 0x20, temp15: 0x21, temp16: 0x22,
         co2_2: 0x10, co2_8: 0x18, h1: 0x33, h2: 0x34, outdoor: 0x04, wc1: 0x13, wc2: 0x12 },
    3: { name: 'typeC', th7: 0x19, temp14: 0x20, temp15: 0x21, temp16: 0x22,
         co2_2: 0x10, co2_8: 0x18, h1: 0x33, h2: 0x34, outdoor: 0x24, wc1: 0x13, wc2: 0x12 },
  };
  if (!configs[type]) throw new Error(`未知场区类型: ${type}`);
  return configs[type];
}
```

### 1.4 验收标准

| 验收项 | 验收标准 |
| :--- | :--- |
| 寄存器可读 | 通过 Modbus TCP 可正确读取 `0x0019` 寄存器值 |
| 默认值 | 未配置时返回 `0` |
| HMI 可配置 | 通过 HMI 可选择场区并保存到 EEPROM |
| 热更新 | 修改场区后无需重启，轮询队列按新配置重建 |
| 断电保存 | 场区配置在断电后保持不变 |

## 二、传感器安装状态寄存器（P0 优先级）

### 2.1 需求背景

测试系统在读取传感器数据前，必须确认环控器中对应传感器已配置为"已安装"。当前 `sensor_setup` 仅能通过 ATE TCP `properties.get` 读取，Modbus TCP 寄存器中无对应映射。为统一通信方式，需将传感器安装状态暴露到 Modbus 寄存器。

### 2.2 固件改动要求

#### 2.2.1 新增传感器安装状态寄存器

在 `BLOCK_CFG`（地址 `0x7001 ~ 0x7009`）之后新增传感器安装状态位掩码寄存器。

**建议寄存器地址**：

| 寄存器地址 | 变量名 | 类型 | 读写 | 说明 |
| :---: | :--- | :---: | :---: | :--- |
| `0x700A` | `SENSOR_TEMP_INSTALLED` | uint16 | RO | 室内温度传感器安装位掩码，bit0~bit15 对应 1#~16# |
| `0x700B` | `SENSOR_HUMI_INSTALLED` | uint16 | RO | 室内湿度传感器安装位掩码，bit0~bit15 对应 1#~16# |
| `0x700C` | `SENSOR_CO2_INSTALLED` | uint8 | RO | 室内CO2传感器安装位掩码，bit0~bit7 对应 1#~8# |
| `0x700D` | `SENSOR_NH3_INSTALLED` | uint8 | RO | 室内氨气传感器安装位掩码，bit0~bit3 对应 1#~4# |
| `0x700E` | `SENSOR_PRESS_INSTALLED` | uint8 | RO | 室内压差传感器安装位掩码，bit0~bit3 对应 1#~4# |
| `0x700F` | `SENSOR_OUTDOOR_INSTALLED` | uint8 | RO | 室外传感器安装位掩码，bit0=室外温湿度，bit1~bit4=水帘1~4 |

**位掩码说明**：
- 每个 bit 对应一路传感器：`1` = 已安装，`0` = 未安装
- 与固件内部 `sensor_setup` / `_deployment_s` 保持一致

#### 2.2.2 Modbus 读取示例

```
请求: 01 03 70 0A 00 06 XX XX
      │  │     │     │
      │  │     │     └── 读取 6 个寄存器（0x700A~0x700F）
      │  │     └── 起始地址 0x700A
      │  └── 功能码 0x03
      └── 从站地址 0x01

响应: 01 03 0C [temp_mask] [humi_mask] [co2_mask] [nh3_mask] [press_mask] [outdoor_mask] XX XX
      │  │  │                                                               │
      │  │  └── 字节数: 12（6 个 uint16/uint8 寄存器）                       │
      │  └── 功能码 0x03                                                     │
      └── 从站地址 0x01                                                      │
                                                                            CRC16
```

**后端解析伪代码**：

```javascript
// 一次读取所有传感器安装状态
const result = await client.readHoldingRegisters(0x700A, 6);
const data = result.data;

const sensorInstalled = {
  temp:   Array.from({length: 16}, (_, i) => Boolean((data[0] >> i) & 1)),
  humi:   Array.from({length: 16}, (_, i) => Boolean((data[1] >> i) & 1)),
  co2:    Array.from({length: 8},  (_, i) => Boolean((data[2] >> i) & 1)),
  nh3:    Array.from({length: 4},  (_, i) => Boolean((data[3] >> i) & 1)),
  press:  Array.from({length: 4},  (_, i) => Boolean((data[4] >> i) & 1)),
  outdoor: Array.from({length: 5}, (_, i) => Boolean((data[5] >> i) & 1)),
};

// 使用示例
if (sensorInstalled.temp[0]) {
  // 1#温度传感器已安装，执行抄读测试
} else {
  // 未安装，跳过或执行未安装屏蔽断言
}
```

### 2.3 验收标准

| 验收项 | 验收标准 |
| :--- | :--- |
| 寄存器可读 | 通过 Modbus TCP 可正确读取 `0x700A~0x700F` |
| 与固件一致 | 返回的安装位掩码与 `sensor_setup` / `_deployment_s` 完全一致 |
| 实时更新 | 通过 HMI 修改传感器安装配置后，寄存器值立即更新 |
| 断电保存 | 安装配置在断电后保持不变 |

## 三、热更新配置寄存器（P0 优先级）

### 3.1 需求背景

传感器测试中需要验证配置热更新功能（启用/禁用传感器、修改告警阈值、修改补偿值等），当前这些配置只能通过 HMI 操作，Modbus TCP 寄存器中无对应映射。自动化测试要求所有配置通过 Modbus TCP 写入。

### 3.2 新增寄存器

在 `BLOCK_CFG`（0x7001~0x700F 之后）新增以下热更新配置寄存器：

| 寄存器地址 | 变量名 | 类型 | 读写 | 说明 |
| :---: | :--- | :---: | :---: | :--- |
| `0x7010` | `RS485_PORT_BASE` | uint16 | R/W | RS485 端口配置基址，0x7010+i 对应第 i+1 路传感器端口号（6/7/8） |
| `0x7040` | `TEMP_HIGH_LIMIT` | int16 | R/W | 温度高限告警阈值，val/10 → ℃ |
| `0x7041` | `TEMP_LOW_LIMIT` | int16 | R/W | 温度低限告警阈值，val/10 → ℃ |
| `0x7042` | `HUMI_HIGH_LIMIT` | uint16 | R/W | 湿度高限告警阈值，val/10 → %RH |
| `0x7043` | `HUMI_LOW_LIMIT` | uint16 | R/W | 湿度低限告警阈值，val/10 → %RH |
| `0x7050~0x705F` | `TEMP_COMP[0..15]` | int16 | R/W | 1#~16# 温度补偿值，val/10 → ℃ |
| `0x7060~0x706F` | `HUMI_COMP[0..15]` | int16 | R/W | 1#~16# 湿度补偿值，val/10 → %RH |
| `0x7070~0x707F` | `PORT_CONFIG[0..15]` | uint16 | R/W | 1#~16# 传感器 RS485 端口配置 |

### 3.3 热更新要求

- 写入后立即生效，无需重启
- 轮询队列（传感器启用/禁用）在写入后自动重建
- 告警阈值在下一个 `Alarm_Check()` 周期生效
- 补偿值在下一次传感器采集时生效
- 断电后保持不变（保存到 EEPROM/Flash）

### 3.4 验收标准

| 验收项 | 验收标准 |
| :--- | :--- |
| 写入回读一致 | writeHoldingRegister 后 readHoldingRegisters 返回相同值 |
| 热更新生效 | 写入后无需重启，对应功能立即按新配置执行 |
| 断电保存 | 断电重启后配置值保持不变 |
| 轮询队列重建 | 传感器启用/禁用后轮询队列自动重建 |

## 四、告警状态寄存器（P0 优先级）

### 4.1 需求背景

传感器异常过滤测试（T-ABNF-001 ErRead、T-ABNF-002 ErMax）和配置热更新测试（T-HOT-004/005 阈值告警）需要通过 Modbus TCP 读取环控器的告警状态。当前告警状态仅在固件内部变量中，未暴露到 Modbus 寄存器。

### 4.2 新增寄存器

| 寄存器地址 | 变量名 | 类型 | 读写 | 说明 |
| :---: | :--- | :---: | :---: | :--- |
| `0x7030` | `ER_READ_FLAG` | uint16 | RO | ErRead 告警标志，bit0~bit15 对应传感器路数 |
| `0x7031` | `ER_MAX_FLAG` | uint16 | RO | ErMax 告警标志，bit0~bit15 对应传感器路数 |
| `0x7032` | `TEMP_HIGH_ALARM` | uint16 | RO | 温度高限告警标志，bit0~bit15 对应传感器路数 |
| `0x7033` | `HUMI_HIGH_ALARM` | uint16 | RO | 湿度高限告警标志，bit0~bit15 对应传感器路数 |
| `0x7034` | `ONLINE_STATUS` | uint16 | RO | 传感器在线状态，bit0~bit15 对应传感器路数（1=在线, 0=离线） |

### 4.3 映射关系

与固件内部变量的对应关系：

| 寄存器 | 固件变量 |
| :--- | :--- |
| `0x7030` | `Sensor_Data.Data_error_Counter` 相关标志（连续 10 次失败置位） |
| `0x7031` | `Sensor_Data.Data_Invariant_Counter` 相关标志（连续 100 次不变置位） |
| `0x7032` | `Alarm_Check()` 温度高限告警结果 |
| `0x7033` | `Alarm_Check()` 湿度高限告警结果 |
| `0x7034` | `Sensor_Data._deployment_s` 在线位 |

### 4.4 验收标准

| 验收项 | 验收标准 |
| :--- | :--- |
| ErRead 触发 | 连续 10 次通信失败后，对应 bit 置 1 |
| ErRead 恢复 | `clearFault` 后对应 bit 清 0 |
| ErMax 触发 | 连续 100 次数值不变后，对应 bit 置 1 |
| 告警触发 | 温度/湿度超过阈值后，对应 bit 置 1 |
| 告警恢复 | 温度/湿度恢复后，按恢复策略清 0 |
| 在线位 | 传感器正常时 bit=1，ErRead 触发后 bit=0 |

## 五、历史缓冲调试寄存器（P0 优先级）

### 5.1 需求背景

历史数据回退测试（T-HIST-001、SEN-HIST-BOOT-001）需要自动化读取和清空历史缓冲。当前历史缓冲只能通过 MSH `sensor_history` 命令手动查看，无法被测试脚本自动化调用。

**关键需求**：
- 读取最近 N 条历史数据（`tm_hour`、`temp`、`humi`、`timestamp`）
- 清空历史缓冲（测试前避免旧数据污染）
- 以上操作必须通过 Modbus TCP 完成，不能依赖 MSH 手动操作

### 5.2 新增寄存器

#### 5.2.1 历史缓冲读取寄存器（0x7100~0x7107）

| 寄存器地址 | 变量名 | 类型 | 读写 | 说明 |
| :---: | :--- | :---: | :---: | :--- |
| `0x7100` | `HISTORY_TOTAL` | uint16 | RO | 历史缓冲总槽位数（固定 25） |
| `0x7101` | `HISTORY_LATEST_IDX` | uint16 | RO | 最新条目索引 |
| `0x7102` | `HISTORY_READ_TRIGGER` | uint16 | WO | 写入索引号触发读取 |
| `0x7103` | `HISTORY_RESULT_HOUR` | uint16 | RO | 读取结果：tm_hour |
| `0x7104` | `HISTORY_RESULT_TEMP` | int16 | RO | 读取结果：温度原始值，val/10 → ℃ |
| `0x7105` | `HISTORY_RESULT_HUMI` | uint16 | RO | 读取结果：湿度原始值，val/10 → %RH |
| `0x7106` | `HISTORY_RESULT_TS_H` | uint16 | RO | 读取结果：timestamp 高 16 位 |
| `0x7107` | `HISTORY_RESULT_TS_L` | uint16 | RO | 读取结果：timestamp 低 16 位 |

#### 5.2.2 历史缓冲清空寄存器（0x7110~0x7111）

| 寄存器地址 | 变量名 | 类型 | 读写 | 说明 |
| :---: | :--- | :---: | :---: | :--- |
| `0x7110` | `HISTORY_CLEAR_TRIGGER` | uint16 | WO | 写 `0xAA55` 触发清空 |
| `0x7111` | `HISTORY_CLEAR_RESULT` | uint16 | RO | 清空结果：0=成功, 非0=失败 |

### 5.3 固件实现建议

```c
// === 历史缓冲读取逻辑 ===

// 当测试系统写入 0x7102 时，固件从 history_InTemp/history_InHumi/time_stamp
// 数组中读取对应索引的数据，写入 0x7103~0x7107

void on_history_read_trigger(uint16_t index) {
    if (index >= HISTORY_LOG_MAX) {
        // 索引越界，返回 0
        return;
    }

    // 注意：history_InTemp 使用环形缓冲区，需要计算实际索引
    // history_InTemp[HISTORY_LOG_MAX - 1] 是最新条目
    // 需要根据实际缓冲区实现计算

    struct tm time_data;
    localtime_r(&time_stamp[index], &time_data);

    modbus_registers[0x7103 - BASE] = time_data.tm_hour;
    modbus_registers[0x7104 - BASE] = (int16_t)(history_InTemp[index] * 10);
    modbus_registers[0x7105 - BASE] = (uint16_t)(history_InHumi[index] * 10);
    modbus_registers[0x7106 - BASE] = (uint16_t)(time_stamp[index] >> 16);
    modbus_registers[0x7107 - BASE] = (uint16_t)(time_stamp[index] & 0xFFFF);
}

// === 历史缓冲清空逻辑 ===

void on_history_clear_trigger(uint16_t magic) {
    if (magic == 0xAA55) {
        // 清空历史缓冲
        memset(history_InTemp, 0, sizeof(history_InTemp));
        memset(history_InHumi, 0, sizeof(history_InHumi));
        memset(time_stamp, 0, sizeof(time_stamp));
        // 重置索引
        history_index = 0;
        modbus_registers[0x7111 - BASE] = 0;  // 成功
    } else {
        modbus_registers[0x7111 - BASE] = 1;  // 魔数错误
    }
}
```

### 5.4 读取时序

```
测试系统                          固件
   |                               |
   |-- 写 0x7102 = 索引号 -------->|  (触发读取)
   |                               |-- 从 history 数组取数据
   |                               |-- 写入 0x7103~0x7107
   |-- 读 0x7103~0x7107 --------->|  (获取结果)
   |                               |
   |-- 写 0x7102 = 下一个索引 ---->|  (读取下一条)
   |                               |
```

### 5.5 验收标准

| 验收项 | 验收标准 |
| :--- | :--- |
| 读取成功 | 写入索引后，0x7103~0x7107 返回正确的历史数据 |
| tm_hour 正确 | 跨小时冻结后，tm_hour 与预期一致（如 11、15、19） |
| temp/humi 正确 | 历史温度/湿度与冻结时的模拟器值一致（误差 ±0.2） |
| timestamp 正确 | 时间戳为冻结时的系统时间 |
| 清空成功 | 写入 0xAA55 后，0x7111 返回 0 |
| 清空后为空 | 清空后读取任意索引，返回值全为 0 |
| 环形缓冲区 | 索引 0~24 循环，最新条目在 0x7101 返回的索引处 |
