## 一、开发目标

将“传感器异常历史数据回退测试方案”落地为自动化测试能力，验证环控器在启动阶段传感器无效时，能够按当前小时从历史缓冲中匹配正确温湿度，并回退到 `ActualTemp/ActualHumi`。

P1 阶段只开发和执行用例 A：

```text
固定模拟值 -> 跨小时冻结历史 -> 重启后保持传感器异常 -> 对时到目标小时 -> 断言启动回退值
```

用例 B（运行时回退）只保留为后续扩展参考，不纳入本任务列表的 P1 必测范围。

---

## 二、核心验收场景

| 组号 | 冻结时间 | 验证时间 | 模拟温度 | 模拟湿度 | 期望历史小时 | 启动回退期望 |
| :---: | :--- | :--- | :---: | :---: | :---: | :--- |
| 1 | 昨天 10:57 | 今天 11:05 | 20.0℃ | 60.0%RH | 11 | `ActualTemp≈20.0`，`ActualHumi≈60.0` |
| 2 | 昨天 14:57 | 今天 15:05 | 22.0℃ | 62.0%RH | 15 | `ActualTemp≈22.0`，`ActualHumi≈62.0` |
| 3 | 昨天 18:57 | 今天 19:05 | 24.0℃ | 64.0%RH | 19 | `ActualTemp≈24.0`，`ActualHumi≈64.0` |

通过标准：

- 历史缓冲确实写入 3 组固定模拟值。
- 重启后传感器仍保持异常，不能恢复实时采样。
- 启动回退按 `tm_hour=11/15/19` 匹配历史条目。
- 断言使用精确值比较，不能只判断“非 0 且非 INVALID_VALUE”。
- 测试结束恢复真实时间，并清除模拟器故障。

### 3.1 详细任务分解

#### HIST-P1-001 场景定义：`SEN-HIST-BOOT-001`

**目标**：把历史回退测试的 3 组冻结/验证数据固化为可执行场景，供 `TestManager` 直接加载。

**建议文件**：`backend/ate/TestScenarioCatalog.js`

**开发内容**：

1. 新增场景 ID：`SEN-HIST-BOOT-001`。
2. 场景类型设置为：`sensor-history-boot-fallback`。
3. 明确执行模式：`executeMode: 'caseAOnly'`。
4. 配置误差范围：`tolerance: 0.2`。
5. 配置 3 组 `freezeGroups`：
   - `group-1`: `freezeHour=10`，`verifyHour=11`，`temp=20.0`，`humi=60.0`
   - `group-2`: `freezeHour=14`，`verifyHour=15`，`temp=22.0`，`humi=62.0`
   - `group-3`: `freezeHour=18`，`verifyHour=19`，`temp=24.0`，`humi=64.0`
6. 配置传感器点位 key：`temp_1`、`humi_1`。
7. 配置场景结束后的恢复动作：`clearFault`、`restoreRealTime`。

**输入**：场景 ID。  
**输出**：完整场景对象。

**异常处理**：

- 找不到场景 ID 时返回明确错误：`SCENARIO_NOT_FOUND`。
- `executeMode` 不是 `caseAOnly` 时拒绝执行，避免误跑用例 B。

**验收标准**：

- `TestManager.loadScenario('SEN-HIST-BOOT-001')` 能返回完整场景。
- 3 组数据完整，不允许缺少 `freezeHour/verifyHour/temp/humi`。
- 场景中明确标注 P1 只执行用例 A。

**交付物**：

- 场景配置代码。
- 场景加载单元测试或 mock 测试。

#### HIST-P1-002 模拟器固定温湿度返回

**目标**：让 PC 端传感器模拟器能按场景固定返回温湿度，作为历史冻结的数据源。

**建议文件**：`backend/ate/SensorSimulator.js`

**开发内容**：

1. 实现或确认 `setSensorValue(key, value)`。
2. 支持 `temp_1`、`humi_1` 两个 key。
3. 将工程值转换为寄存器原始值，例如温度/湿度按 `value * 10` 存储。
4. 更新影子寄存器表，记录 `rawValue/value/unit/updatedAt/scenarioId`。
5. 环控器发起 RS485 抄读时，模拟器从影子寄存器取值回复。
6. 记录交易日志：请求帧、响应帧、解析值、场景 ID。

**输入**：

```js
await sensorSimulator.setSensorValue('temp_1', 20.0)
await sensorSimulator.setSensorValue('humi_1', 60.0)
```

**输出**：

- 影子寄存器更新成功。
- 环控器抄读时收到对应温湿度。

**异常处理**：

- key 不存在时返回 `SENSOR_KEY_NOT_FOUND`。
- value 超出合法范围时返回 `SENSOR_VALUE_OUT_OF_RANGE`。
- 串口未启动时返回 `SIMULATOR_NOT_STARTED`。

**验收标准**：

- 20.0℃/60.0%RH、22.0℃/62.0%RH、24.0℃/64.0%RH 三组值均能稳定返回。
- 交易日志能看到对应请求和响应。
- 同一场景内多次抄读值保持一致，直到下一次 `setSensorValue()`。

**交付物**：

- 固定值写入接口。
- 影子寄存器查询接口或日志。
- mock 测试：设置值后读取影子寄存器。

#### HIST-P1-003 模拟器重启后持续异常

**目标**：确保用例 A 重启环控器后，传感器仍然异常，避免环控器读取到实时值导致误判。

**建议文件**：`backend/ate/SensorSimulator.js`

**开发内容**：

1. 扩展 `injectTimeout({ key, persist })`。
2. 当 `persist=true` 时，故障状态保存在模拟器内存状态中，不因环控器断线、重连、重启而自动清除。
3. 故障期间对应 key 不回复，或按测试约定返回无效响应。
4. 实现 `clearFault(key)`，测试结束后恢复正常。
5. 交易日志记录 `fault=timeout`、`persist=true`、未响应次数。

**输入**：

```js
await sensorSimulator.injectTimeout({ key: 'temp_1', persist: true })
await sensorSimulator.injectTimeout({ key: 'humi_1', persist: true })
```

**输出**：

- 对应传感器点位持续不响应。

**异常处理**：

- key 不存在时报错。
- 已经存在故障时允许幂等设置。
- `clearFault()` 后必须恢复正常响应。

**验收标准**：

- 环控器重启前后，`temp_1/humi_1` 都保持异常。
- 只有调用 `clearFault()` 后才恢复。
- 日志能证明重启后仍发生超时。

**交付物**：

- `persist` 故障状态实现。
- 故障状态查询或日志。
- 重启保持异常 mock 测试。

#### HIST-P1-004 Modbus TCP 对时接口

**目标**：测试系统能通过 Modbus TCP 设置环控器时间，并确认写入成功。

**建议文件**：`backend/ate/ControllerStateReader.js`

**开发内容**：

1. 实现 `syncTime({ year, month, day, hour, minute, second })`。
2. 写 HR10~HR15。
3. 写 HR16=1 触发对时。
4. 轮询 HR17，直到返回 0 或超时。
5. 读回 HR10~HR15，确认设备时间与目标时间一致。
6. 返回结构化结果。

**输入**：

```js
await controllerStateReader.syncTime({
  year: 2026, month: 6, day: 15,
  hour: 10, minute: 57, second: 0
})
```

**输出**：

```js
{
  ok: true,
  targetTime: '2026-06-15 10:57:00',
  deviceTime: '2026-06-15 10:57:00'
}
```

**异常处理**：

- TCP 未连接：`MODBUS_TCP_NOT_CONNECTED`。
- HR17 非 0：返回固件错误码。
- 读回时间不一致：`TIME_VERIFY_FAILED`。
- 超时：`TIME_SYNC_TIMEOUT`。

**验收标准**：

- 对时到 10:57、14:57、18:57 均成功。
- 对时后 HR13 与目标小时一致。
- 失败时报告能显示目标时间、读回时间、HR17。

**交付物**：

- `syncTime()`。
- 对时失败重试或超时配置。
- mock 测试。

#### HIST-P1-005 设备重启与重连

**目标**：测试系统能通过 Modbus TCP 重启环控器，并在设备恢复后自动重连。

**建议文件**：`backend/ate/ControllerStateReader.js`

**开发内容**：

1. 实现 `reboot()`。
2. 写 HR18=`0x55AA`。
3. 关闭当前 TCP 连接。
4. 等待设备重启，按间隔重连。
5. 重连成功后读取一个基础寄存器确认设备可用。
6. 返回重启耗时和重连结果。

**异常处理**：

- 写 HR18 失败：`REBOOT_COMMAND_FAILED`。
- 超过最大等待时间仍无法连接：`REBOOT_RECONNECT_TIMEOUT`。
- 重连后读寄存器失败：`DEVICE_NOT_READY`。

**验收标准**：

- 3 组启动回退验证前均能成功重启并重连。
- 重连失败时测试立即 FAIL，并执行清理动作。

**交付物**：

- `reboot()`。
- 重连参数：最大等待时间、重试间隔。
- 重启日志。

#### HIST-P1-006 读取 `ActualTemp/ActualHumi`

**目标**：读取环控器回退后的实际温湿度，作为最终断言对象。

**建议文件**：`backend/ate/ControllerStateReader.js`

**开发内容**：

1. 明确 `ActualTemp/ActualHumi` 的 Modbus TCP 寄存器地址。
2. 实现 `readActualTempHumi()`。
3. 按固件比例系数换算，例如寄存器值 `/10`。
4. 处理有符号值、无效值、异常码。
5. 返回原始值和工程值。

**输出示例**：

```js
{
  actualTempRaw: 200,
  actualTemp: 20.0,
  actualHumiRaw: 600,
  actualHumi: 60.0
}
```

**异常处理**：

- 寄存器地址未配置：`ACTUAL_REGISTER_NOT_CONFIGURED`。
- 读到 `INVALID_VALUE`：返回 `invalid: true`，并保留 raw。
- 换算异常：`VALUE_CONVERT_FAILED`。

**验收标准**：

- 能读取 20.0/60.0、22.0/62.0、24.0/64.0 对应值。
- 断言失败时报告 raw 和 value，方便判断是读取错还是换算错。

**交付物**：

- `readActualTempHumi()`。
- 寄存器地址配置。
- 换算规则测试。

#### HIST-P1-007 读取历史缓冲

**目标**：自动读取历史缓冲，证明跨小时冻结真正写入了预期值。

**建议文件**：`backend/ate/ControllerStateReader.js`

**开发内容**：

1. 实现 `readHistoryTail(count)`。
2. 返回最近 N 条历史数据。
3. 每条数据包含 `index/tm_hour/temp/humi/timestamp/raw`。
4. 支持通过以下任一方式实现：
   - Modbus TCP 调试寄存器。
   - MSH `sensor_history` 输出解析。
   - 测试专用 API。
5. 对返回数据按时间或缓冲区顺序排序。

**输出示例**：

```js
[
  { index: 22, tm_hour: 11, temp: 20.0, humi: 60.0, timestamp: 1781456400 },
  { index: 23, tm_hour: 15, temp: 22.0, humi: 62.0, timestamp: 1781470800 },
  { index: 24, tm_hour: 19, temp: 24.0, humi: 64.0, timestamp: 1781485200 }
]
```

**异常处理**：

- 固件无读取接口：任务阻塞，不能用“读 HR13 成功”替代。
- 解析 MSH 失败：输出原始文本和解析错误。
- 历史值为 0 或 INVALID：返回数据，但断言失败。

**验收标准**：

- `readHistoryTail(1)` 能在每次跨小时后读到最新条目。
- `readHistoryTail(3)` 能读到三组固定值。
- 历史条目必须包含 `tm_hour`，否则不能判断启动回退匹配来源。

**交付物**：

- `readHistoryTail()`。
- 历史数据解析器。
- 历史快照报告字段。

#### HIST-P1-008 清空历史缓冲

**目标**：测试前清除旧历史数据，避免 FlashDB 持久化旧值污染本轮断言。

**建议文件**：`backend/ate/ControllerStateReader.js`，必要时需要固件配合。

**开发内容**：

1. 实现 `clearHistory()`。
2. 支持以下任一方式：
   - MSH `sensor_history_clear`。
   - Modbus TCP 调试寄存器触发清空。
   - 测试 API。
3. 清空后调用 `readHistoryTail(3)` 验证结果。
4. 如果固件不允许清空，至少记录测试前历史快照并在报告中标记风险。

**异常处理**：

- 清空接口不存在：`HISTORY_CLEAR_NOT_SUPPORTED`，标记阻塞或降级。
- 清空后仍有旧数据：`HISTORY_CLEAR_VERIFY_FAILED`。

**验收标准**：

- 测试前历史缓冲可清空，或能明确证明旧数据不会影响本轮 3 组断言。
- 清空动作写入报告。

**交付物**：

- `clearHistory()`。
- 清空确认逻辑。

#### HIST-P1-009 冻结阶段编排

**目标**：自动完成三组历史冻结，并验证历史缓冲写入正确。

**建议文件**：`backend/ate/TestManager.js`

**开发内容**：

1. 加载 `SEN-HIST-BOOT-001`。
2. 执行 `clearHistory()`。
3. 对每个 freezeGroup：
   - 设置模拟器固定温湿度。
   - 对时到昨天 `freezeHour:57:00`。
   - 等待跨小时到 `verifyHour`。
   - 读取最新历史。
   - 断言 `tm_hour/temp/humi`。
4. 冻结结束后读取最近 3 条历史快照。
5. 将每组冻结结果写入上下文，供启动回退阶段使用。

**异常处理**：

- 任一组冻结失败，停止后续验证并进入恢复阶段。
- 对时跳变导致额外历史写入时，标记 `HISTORY_POLLUTED`。
- 历史值不匹配时，报告期望值、实际值、差值。

**验收标准**：

- 能自动完成 3 组冻结。
- 不能只依赖 HR13 判断成功。
- 失败时能定位是哪一组冻结失败。

**交付物**：

- `runHistoryFreezePhase(scenario)` 或等价函数。
- 冻结阶段报告。

#### HIST-P1-010 用例 A 启动回退编排

**目标**：自动完成三组启动回退验证。

**建议文件**：`backend/ate/TestManager.js`

**开发内容**：

1. 对每个 freezeGroup：
   - 设置 `temp_1/humi_1` 持续异常。
   - 重启环控器。
   - 对时到今天 `verifyHour:05:00`。
   - 等待回退逻辑执行。
   - 读取 `ActualTemp/ActualHumi`。
   - 断言回退值等于 freezeGroup 的固定值。
2. 每组执行前后保存交易日志。
3. 任意失败时进入恢复阶段。

**异常处理**：

- 重启后模拟器故障丢失：`FAULT_NOT_PERSISTED`。
- 回退值为实时采样值：标记启动异常保持失败。
- 回退值非预期历史值：标记 `BOOT_FALLBACK_VALUE_MISMATCH`。

**验收标准**：

- 今天 11:05 回退到 20.0℃/60.0%RH。
- 今天 15:05 回退到 22.0℃/62.0%RH。
- 今天 19:05 回退到 24.0℃/64.0%RH。

**交付物**：

- `runHistoryBootFallbackPhase(scenario)` 或等价函数。
- 三组启动回退报告。

#### HIST-P1-011 断言引擎

**目标**：统一处理历史测试中的精确断言和失败原因。

**建议文件**：`backend/ate/AssertEngine.js`

**开发内容**：

1. 实现 `assertClose(actual, expected, tolerance, message)`。

## 四、接口开发任务明细

### 4.1 `SensorSimulator`

建议文件：`backend/ate/SensorSimulator.js`

必须新增或确认以下接口：

```js
await sensorSimulator.setSensorValue('temp_1', 20.0)
await sensorSimulator.setSensorValue('humi_1', 60.0)

await sensorSimulator.injectTimeout({ key: 'temp_1', persist: true })
await sensorSimulator.injectTimeout({ key: 'humi_1', persist: true })

await sensorSimulator.clearFault('temp_1')
await sensorSimulator.clearFault('humi_1')
```

开发要求：

- `setSensorValue()` 写入影子寄存器，环控器下一次 RS485 抄读时返回对应值。
- `persist: true` 表示故障状态不因环控器重启而自动清除。
- `clearFault()` 必须能恢复正常响应。
- 交易日志需要记录每次请求/响应、故障状态、场景 ID。

完成标准：

- 冻结阶段能稳定返回 3 组固定温湿度。
- 启动回退阶段能持续不响应，直到测试结束主动清除故障。

### 4.2 `ControllerStateReader`

建议文件：`backend/ate/ControllerStateReader.js`

必须实现：

```js
await controllerStateReader.syncTime({
  year,
  month,
  day,
  hour,
  minute,
  second
})

await controllerStateReader.reboot()
await controllerStateReader.clearHistory()
const history = await controllerStateReader.readHistoryTail(3)
const actual = await controllerStateReader.readActualTempHumi()
await controllerStateReader.restoreRealTime()
```

寄存器约定：

| 功能 | 寄存器 | 说明 |
| :--- | :--- | :--- |
| 年月日时分秒 | HR10~HR15 | Modbus TCP 写入目标时间 |
| 对时触发 | HR16 | 写 1 触发对时 |
| 对时结果 | HR17 | 读 0 表示成功 |
| 设备重启 | HR18 | 写 `0x55AA` 触发重启 |

待确认项：

- `ActualTemp/ActualHumi` 的 Modbus TCP 寄存器地址。
- 历史缓冲读取方式：调试寄存器、MSH `sensor_history` 输出解析、或测试 API。
- 历史缓冲清空方式：`sensor_history_clear`、调试寄存器、或测试 API。

完成标准：

- `syncTime()` 写入后必须读回 HR10~HR17 确认。
- `reboot()` 后自动等待并重连。
- `readHistoryTail(3)` 至少返回 `tm_hour/temp/humi/timestamp`。
- 没有历史读取接口时，该任务不能标记完成。

## 五、测试编排任务

### 5.1 冻结阶段

执行流程：

```text
clearHistory()
for each freezeGroup:
  setSensorValue(temp_1, group.temp)
  setSensorValue(humi_1, group.humi)
  syncTime(yesterday group.freezeHour:57:00)
  wait until HR13 == group.verifyHour
  history = readHistoryTail(1)
  assert history.latest.tm_hour == group.verifyHour
  assertClose(history.latest.temp, group.temp)
  assertClose(history.latest.humi, group.humi)
readHistoryTail(3)
assert 3 groups exist
```

完成标准：

- 3 组冻结历史均能在历史缓冲中查到。
- 任何一组历史值不匹配，立即 FAIL。
- 不能只用 HR13 跨小时成功作为冻结成功依据。

### 5.2 用例 A 启动回退阶段

执行流程：

```text
for each freezeGroup:
  injectTimeout(temp_1, persist=true)
  injectTimeout(humi_1, persist=true)
  reboot()
  syncTime(today group.verifyHour:05:00)
  wait fallback stable
  actual = readActualTempHumi()
  assertClose(actual.actualTemp, group.temp)
  assertClose(actual.actualHumi, group.humi)
```

完成标准：

- 重启后传感器仍异常。
- `ActualTemp/ActualHumi` 分别回退到 3 组冻结值。
- 报告中能看到每组的期望值、实际值和误差。

### 5.3 恢复阶段

执行流程：

```text
clearFault(temp_1)
clearFault(humi_1)
restoreRealTime()
read back HR10~HR15
```

完成标准：

- 模拟器恢复正常响应。
- 设备时间恢复为当前真实时间。
- 即使测试失败，也要执行恢复动作。
