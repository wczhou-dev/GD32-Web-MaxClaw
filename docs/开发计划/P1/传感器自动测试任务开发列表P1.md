## 一、开发目标

将 `传感器自动测试内容开发清单P1.md` 中的全部 P1 传感器自动测试内容落地为可开发、可验收、可自动执行的任务列表。测试系统需要完成从传感器模拟器输入、环控器 RS485 抄读、Modbus TCP 读取环控器处理结果、断言判定、报告输出到前端展示的完整闭环。

历史数据回退部分只开发和执行用例 A：

```text
固定模拟值 -> 跨小时冻结历史 -> 重启后保持传感器异常 -> 对时到目标小时 -> 断言启动回退值
```

用例 B（运行时回退）只保留为后续扩展参考，不纳入本任务列表的 P1 必测范围。

### 3.8 P1 全量非历史任务详细卡

#### BASE-P1-001 场区适配

- 任务类型：接口设计/测试前置
- 目标说明：测试开始前读取环控器场区类型，加载正确的传感器地址表，避免模拟器地址与环控器抄读地址不一致。
- 涉及模块：`ControllerStateReader`、`SensorSimulator`、场区配置
- 硬件/资源依赖：环控器 Modbus TCP；无硬件时可 mock `0x0019`
- 可能涉及文件：
  - `backend/ate/ControllerStateReader.js`
  - `backend/ate/SensorSimulator.js`
  - `backend/ate/fieldConfigs/*.js`
- 前置依赖：无
- 输入条件：
  - Modbus TCP 可读取 `0x0019`
  - 场区编码：0 未配置，1 类型 A，2 类型 B，3 类型 C
- 期望输出：
  - 当前场区类型
  - 对应场区传感器地址表
  - 模拟器加载同一地址表
- 验收标准：
  - [ ] 读到 0 时阻止测试并提示先配置场区
  - [ ] 类型 A/B/C 均能加载对应地址表
  - [ ] 模拟器当前场区与环控器一致
- 验证方式：
  - [ ] mock 0/1/2/3 四种返回值
  - [ ] 实机读取 `0x0019`
- 风险点：场区地址表错误会导致所有 RS485 抄读类测试失败
- 安全释放要求：无物理动作；失败时不启动后续测试
- 任务规模：S

#### BASE-P1-002 传感器安装确认

- 任务类型：测试前置/状态读取
- 目标说明：读取传感器安装状态，决定测试项进入正常测试还是未安装屏蔽断言。
- 涉及模块：`ControllerStateReader`、`TestManager`
- 硬件/资源依赖：环控器 Modbus TCP
- 可能涉及文件：
  - `backend/ate/ControllerStateReader.js`
  - `backend/ate/TestManager.js`
- 前置依赖：BASE-P1-001
- 输入条件：
  - `readHoldingRegisters(0x700A, 6)`
- 期望输出：
  - `temp[16]`、`humi[16]`、`co2[8]`、`nh3[4]`、`press[4]`、`outdoor[5]` 安装状态数组
- 验收标准：
  - [ ] 安装位解析正确
  - [ ] 未安装传感器不跳过，而是进入 `not_installed_masked` 断言
  - [ ] 报告记录未安装原因
- 验证方式：
  - [ ] mock 位图解析
  - [ ] 实机读取 `0x700A~0x700F`
- 风险点：位序解析错误会造成误测或漏测
- 安全释放要求：无
- 任务规模：S

#### READ-P1-001 室内温度抄读

- 任务类型：测试用例
- 目标说明：验证 16 路室内温度逐路采集和 `ActualTemp` 平均值计算。
- 涉及模块：`SensorSimulator`、`ControllerStateReader`、`AssertEngine`
- 硬件/资源依赖：USB-RS485、环控器；mock 阶段无
- 可能涉及文件：
  - `backend/ate/TestScenarioCatalog.js`
  - `backend/ate/TestManager.js`
- 前置依赖：BASE-P1-001~006、SYS-P1-001~004
- 输入条件：
  - 16 路温度模拟值，如 20.0℃~35.0℃
  - 对应温度传感器安装位为已安装
- 期望输出：
  - `0x1001 + i*2` 逐路温度等于模拟值
  - `ActualTemp`（寄存器 `0x103B`，已避开 CO2 寄存器冲突）等于已安装有效温度平均值
- 验收标准：
  - [ ] 逐路温度误差 <= ±0.1℃
  - [ ] ActualTemp 误差 <= ±0.1℃
  - [ ] 未安装路进入屏蔽断言
- 验证方式：
  - [ ] mock 寄存器读取

## 四、历史回退核心验收场景

| 组号 | 冻结时间 | 验证时间 | 模拟温度 | 模拟湿度 | 期望历史小时 | 启动回退期望 |
| :---: | :--- | :--- | :---: | :---: | :---: | :--- |
| 1 | 昨天 10:57 | 今天 11:05 | 20.0℃ | 60.0%RH | 11 | `ActualTemp≈20.0`，`ActualHumi≈60.0` |
| 2 | 昨天 14:57 | 今天 15:05 | 22.0℃ | 62.0%RH | 15 | `ActualTemp≈22.0`，`ActualHumi≈62.0` |
| 3 | 昨天 18:57 | 今天 19:05 | 24.0℃ | 64.0%RH | 19 | `ActualTemp≈24.0`，`ActualHumi≈64.0` |

通过标准：

- 历史缓冲确实写入 3 组固定模拟值。
- 重启后传感器仍保持异常，不能恢复实时采样。
- 启动回退按 `tm_hour=11/15/19` 匹配历史条目。
- 断言使用精确值比较，不能只判断“非 0 且非 INVALID_VALUE”。
- 测试结束恢复真实时间，并清除模拟器故障。

#### HIST-P1-002 模拟器固定温湿度返回

**目标**：让 PC 端传感器模拟器能按场景固定返回温湿度，作为历史冻结的数据源。

**建议文件**：`backend/ate/SensorSimulator.js`

**开发内容**：

1. 实现或确认 `setSensorValue(key, value)`。
2. 支持 `temp_1`、`humi_1` 两个 key。
3. 将工程值转换为寄存器原始值，例如温度/湿度按 `value * 10` 存储。
4. 更新影子寄存器表，记录 `rawValue/value/unit/updatedAt/scenarioId`。
5. 环控器发起 RS485 抄读时，模拟器从影子寄存器取值回复。
6. 记录交易日志：请求帧、响应帧、解析值、场景 ID。

**输入**：

```js
await sensorSimulator.setSensorValue('temp_1', 20.0)
await sensorSimulator.setSensorValue('humi_1', 60.0)
```

**输出**：

- 影子寄存器更新成功。
- 环控器抄读时收到对应温湿度。

**异常处理**：

- key 不存在时返回 `SENSOR_KEY_NOT_FOUND`。
- value 超出合法范围时返回 `SENSOR_VALUE_OUT_OF_RANGE`。
- 串口未启动时返回 `SIMULATOR_NOT_STARTED`。

**验收标准**：

- 20.0℃/60.0%RH、22.0℃/62.0%RH、24.0℃/64.0%RH 三组值均能稳定返回。
- 交易日志能看到对应请求和响应。
- 同一场景内多次抄读值保持一致，直到下一次 `setSensorValue()`。

**交付物**：

- 固定值写入接口。
- 影子寄存器查询接口或日志。
- mock 测试：设置值后读取影子寄存器。

#### HIST-P1-003 模拟器重启后持续异常

**目标**：确保用例 A 重启环控器后，传感器仍然异常，避免环控器读取到实时值导致误判。

**建议文件**：`backend/ate/SensorSimulator.js`

**开发内容**：

1. 扩展 `injectTimeout({ key, persist })`。
2. 当 `persist=true` 时，故障状态保存在模拟器内存状态中，不因环控器断线、重连、重启而自动清除。
3. 故障期间对应 key 不回复，或按测试约定返回无效响应。
4. 实现 `clearFault(key)`，测试结束后恢复正常。
5. 交易日志记录 `fault=timeout`、`persist=true`、未响应次数。

**输入**：

```js
await sensorSimulator.injectTimeout({ key: 'temp_1', persist: true })
await sensorSimulator.injectTimeout({ key: 'humi_1', persist: true })
```

**输出**：

- 对应传感器点位持续不响应。

**异常处理**：

- key 不存在时报错。
- 已经存在故障时允许幂等设置。
- `clearFault()` 后必须恢复正常响应。

**验收标准**：

- 环控器重启前后，`temp_1/humi_1` 都保持异常。
- 只有调用 `clearFault()` 后才恢复。
- 日志能证明重启后仍发生超时。

**交付物**：

- `persist` 故障状态实现。
- 故障状态查询或日志。
- 重启保持异常 mock 测试。

#### HIST-P1-004 Modbus TCP 对时接口

**开发内容**：

```js
await controllerStateReader.syncTime({
```

- 超时：`TIME_SYNC_TIMEOUT`。

**验收标准**：

- 对时到 10:57、14:57、18:57 均成功。

#### HIST-P1-006 读取 `ActualTemp/ActualHumi`

**目标**：读取环控器回退后的实际温湿度，作为最终断言对象。

**建议文件**：`backend/ate/ControllerStateReader.js`

**开发内容**：

1. 明确 `ActualTemp/ActualHumi` 的 Modbus TCP 寄存器地址为 `0x103B` 与 `0x103C`（已避开 CO2 占用冲突）。
2. 实现 `readActualTempHumi()`。
3. 按固件比例系数换算，例如寄存器值 `/10`。
4. 处理有符号值、无效值、异常码。
5. 返回原始值和工程值。

**输出示例**：

```js
{
  actualTempRaw: 200,
  actualTemp: 20.0,
  actualHumiRaw: 600,
  actualHumi: 60.0

```

**异常处理**：

**验收标准**：

- 能读取 20.0/60.0、22.0/62.0、24.0/64.0 对应值。
- 断言失败时报告 raw 和 value，方便判断是读取错还是换算错。

**交付物**：

- `readActualTempHumi()`。
- 寄存器地址配置。
- 换算规则测试。

#### HIST-P1-007 读取历史缓冲

4. 支持通过以下任一方式实现：
   - Modbus TCP 调试寄存器。

**输出示例**：

```js
[
  { index: 22, tm_hour: 11, temp: 20.0, humi: 60.0, timestamp: 1781456400 },
```

## 六、历史回退接口开发任务明细

### 6.1 `SensorSimulator`

建议文件：`backend/ate/SensorSimulator.js`

必须新增或确认以下接口：

```js
await sensorSimulator.setSensorValue('temp_1', 20.0)
await sensorSimulator.setSensorValue('humi_1', 60.0)

await sensorSimulator.injectTimeout({ key: 'temp_1', persist: true })
```

开发要求：

- `persist: true` 表示故障状态不因环控器重启而自动清除。
- `clearFault()` 必须能恢复正常响应。

完成标准：

- 冻结阶段能稳定返回 3 组固定温湿度。
- 启动回退阶段能持续不响应，直到测试结束主动清除故障。

### 6.2 `ControllerStateReader`

建议文件：`backend/ate/ControllerStateReader.js`

必须实现：

```js
await controllerStateReader.syncTime({
  year,
  month,
  day,
  hour,
  minute,
  second
})

await controllerStateReader.reboot()
await controllerStateReader.clearHistory()
const history = await controllerStateReader.readHistoryTail(3)
```

寄存器约定：

| 功能 | 寄存器 | 说明 |
| :--- | :--- | :--- |
| 年月日时分秒 | HR10~HR15 | Modbus TCP 写入目标时间 |
| 对时触发 | HR16 | 写 1 触发对时 |
| 对时结果 | HR17 | 读 0 表示成功 |
| 设备重启 | HR18 | 写 `0x55AA` 触发重启 |

- `ActualTemp/ActualHumi` 的 Modbus TCP 寄存器地址。

- 历史缓冲清空方式：`sensor_history_clear`、调试寄存器、或测试 API。

完成标准：

- `syncTime()` 写入后必须读回 HR10~HR17 确认。
- `reboot()` 后自动等待并重连。
- `readHistoryTail(3)` 至少返回 `tm_hour/temp/humi/timestamp`。
- 没有历史读取接口时，该任务不能标记完成。

## 七、历史回退测试编排任务

### 7.1 冻结阶段

执行流程：

```text
clearHistory()
for each freezeGroup:
  setSensorValue(temp_1, group.temp)
  setSensorValue(humi_1, group.humi)
  syncTime(yesterday group.freezeHour:57:00)
  wait until HR13 == group.verifyHour
  history = readHistoryTail(1)
  assert history.latest.tm_hour == group.verifyHour
  assertClose(history.latest.temp, group.temp)
  assertClose(history.latest.humi, group.humi)
readHistoryTail(3)
assert 3 groups exist
```

完成标准：

- 3 组冻结历史均能在历史缓冲中查到。
- 任何一组历史值不匹配，立即 FAIL。
- 不能只用 HR13 跨小时成功作为冻结成功依据。

### 7.2 用例 A 启动回退阶段

执行流程：

```text
for each freezeGroup:
  injectTimeout(temp_1, persist=true)
  injectTimeout(humi_1, persist=true)
  reboot()
  syncTime(today group.verifyHour:05:00)
  wait fallback stable
  actual = readActualTempHumi()
  assertClose(actual.actualTemp, group.temp)
  assertClose(actual.actualHumi, group.humi)
```

完成标准：

- 重启后传感器仍异常。
- `ActualTemp/ActualHumi` 分别回退到 3 组冻结值。
- 报告中能看到每组的期望值、实际值和误差。

### 7.3 恢复阶段

执行流程：

```text
clearFault(temp_1)
clearFault(humi_1)
restoreRealTime()
read back HR10~HR15
```

完成标准：

- 模拟器恢复正常响应。
- 设备时间恢复为当前真实时间。
- 即使测试失败，也要执行恢复动作。

### 13.13 全量风险与待确认问题

| 问题 | 影响 | 建议处理 |
| :--- | :--- | :--- |
| 场区类型寄存器 `0x0019` 是否已固件支持 | 无法自动加载地址表 | 若未支持，先补固件或提供临时配置 |
| 安装状态寄存器位序是否最终确认 | 未安装屏蔽可能误判 | 与固件寄存器文档对齐并加 mock |
| `ActualTemp/ActualHumi` 地址与缩放是否最终确认 | 平均值断言可能错误 | 在 `ControllerStateReader` 集中配置 |
| 历史缓冲读取接口是否存在 | 历史回退无法闭环 | 优先补 `readHistoryTail` 数据源 |
| 告警寄存器和恢复延时是否明确 | 阈值测试可能偶发失败 | 把延时策略写入场景配置 |
| RS485 端口切换硬件资源是否具备 | T-HOT-003 可能无法 HIL | 先做 mock，HIL 标记资源依赖 |
| ErMax 100 次轮询耗时较长 | 批量测试耗时增加 | 减少启用传感器数量或提供测试阈值钩子 |
