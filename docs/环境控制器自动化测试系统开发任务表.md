## 三、任务状态、优先级与开发原则

| 标记 | 含义 | 使用场景 |
| :--- | :--- | :--- |
| `[○]` | 未开始 | 尚未编码或尚未整理交付物 |
| `[◉]` | 正在执行 | 已开始实现或正在联调 |
| `[●]` | 已完成 | 已完成开发、通过验证，可作为依赖 |
| `[◇]` | 联调验收 | 代码主体已存在，不新增开发，只做对接验证、缺陷记录和修正 |
| `[△]` | 可后置 | 不影响第一版主流程，可在核心链路跑通后补强 |

| 优先级 | 含义 |
| :--- | :--- |
| `P0` | 第一版必须完成，否则无法形成可用系统 |
| `P1` | 第一版建议完成，影响效率、可追溯性或现场体验 |
| `P2` | 增强项，可在基础验收通过后迭代 |

开发原则：

1. 不新建独立上位机项目，直接扩展 `GD32-Web-MaxClaw`。
2. 不修改环控器固件主体，固件端只做联调验收和缺陷记录。
3. 不在 Vue 组件中直接写 Modbus、TCP、原始十六进制解析；原始协议逻辑全部在后端或共享常量层。
4. 不在组件中散落寄存器地址、错误码、CmdType、测试项 ID；统一从 `shared/constants.js` 或后端协议模块引用。
5. UI 继续使用 Vue 3 + Element Plus；按钮使用 Element Plus icons，不用 emoji 当功能图标。
6. ATE 是产线工具，不做营销首页；首屏就是可操作的单台检定工作台。
7. 模块化迭代与持续编译提交原则：每完成一个独立模块或一个明确功能闭环后，必须立即执行对应的本地验证，并完成一次阶段性 Git 提交与同步。严禁连续堆积多个未验证模块后统一编译、统一提交；也不允许把前端 UI、后端协议、报告服务、配置文件等无关改动混在同一次提交里。

| 模块类型 | 示例模块 | 完成后必须执行的本地验证 | Git 提交要求 |
| :--- | :--- | :--- | :--- |
| 后端协议模块 | `AteFrameCodec.js`、`AteTcpClient.js`、`AteMessageRouter.js` | 至少执行语法检查、模块级脚本或后端启动验证；协议模块必须覆盖半包、粘包、坏 Magic、长度超限、非法 JSON | 单独提交，示例：`feat(ate-codec): 实现 ATE TCP JSON 帧编解码` |
| 后端业务模块 | `TestManager.js`、`TestCatalog.js`、`TestReportService.js`、`backend/api/test.js` | 执行后端启动验证，并用模拟 session 或报告样例验证核心路径 | 单独提交，提交信息体现业务模块和目的 |
| 前端状态与页面 | `deviceStore.js`、`AteTest.vue`、`App.vue` | 在 `frontend` 执行 `npm run build`，确认无编译错误；关键交互需用页面自测验证 | 单独提交，避免与后端协议改动混合 |
| 共享常量与配置 | `shared/constants.js`、`.env.example`、`backend/config/devices.json` | 确认前后端引用路径一致，常量命名无冲突，默认配置可启动 | 单独提交，说明常量或配置边界 |
| 联调闭环 | 例如 `start_test_request -> TestManager -> AteTcpClient -> test_progress_update` | 完成一次最小链路联调，记录 PASS/FAIL/超时/停止路径 | 一个闭环一个提交，便于回滚和定位 |

## 四、建议公共接口与配置项

### 4.0 `shared/constants.js` 公共常量规划

`shared/constants.js` 是 MaxClaw 前后端协议同步的公共层，必须优先补齐，避免后续在 Vue 组件、后端服务和报告模块中散落魔数。

| 常量类别 | 建议内容 | 说明 |
| :--- | :--- | :--- |
| ATE 寄存器块 | `BLOCK_TEST_STATUS: 0x8000-0x802F`、`BLOCK_TEST_CONFIG: 0x8030-0x803F`、`BLOCK_ENV: 0x1001-0x1030`、`BLOCK_HW: 0x4001-0x4015` | 与 P0 第 3 章和 MaxClaw `docs/modbus-map.md` 对齐 |
| 测试掩码 | `ATE_MASK_ALL = 0x01FF` | 仅允许 bit0~bit8 覆盖 9 项基础硬件自检；下发前必须过滤未定义位 |
| 整体状态码 | `Idle=0`、`Running=1`、`Finished=2`、`Error=3` | 供后端状态机、前端状态栏和报告共同使用 |
| 单项结果码 | `Pending=0`、`Testing=1`、`Pass=2`、`Fail=3`、`Skip=4` | 供项目树、大表行状态和报告 timeline 使用 |
| 固件错误码 Map | 如 Flash 故障 `0x0010`、EEPROM 故障 `0x0020` 及后续错误码 | 每个错误码必须绑定中文原因、排障建议和可选 C 变量上下文 |
| ATE 帧协议 | `MAGIC=0x55AA`、`CMD_DOWNLINK=0x0001`、`CMD_UPLINK=0x0002`、`CMD_ACK=0x0003` | `AteFrameCodec` 和测试日志统一引用 |

### 4.1 WebSocket 消息

前端到后端：

| 消息类型 | 必填字段 | 说明 |
| :--- | :--- | :--- |
| `start_test_request` | `deviceIp`、`operatorInputId`、`selectedItemIds`、`deviceModel`、`workOrder` | 启动一次自动化检定会话 |
| `stop_test_request` | `deviceIp`、`sessionId` | 停止当前会话并触发安全释放 |
| `reset_test_request` | `deviceIp`、`sessionId` | 清空测试状态和 UI 高亮，复位底板测试状态 |
| `retry_failed_request` | `deviceIp`、`sessionId` | 仅重测当前会话失败项 |
| `manual_force_io_request` | `deviceIp`、`outputs`、`timeoutMs` | 手动点检页强制继电器或 AO 输出 |
| `get_test_session` | `deviceIp` | 前端刷新或重连后恢复当前测试会话 |

后端到前端：

| 消息类型 | 字段 | 说明 |
| :--- | :--- | :--- |
| `test_progress_update` | `session`、`currentItem`、`summary` | 实时测试状态、当前项、通过率 |
| `test_finished_notification` | `reportId`、`summary`、`downloadUrl` | 测试结束并生成报告 |
| `test_error` | `deviceIp`、`sessionId`、`errCode`、`message`、`stage` | 启动失败、通信异常、ACK 超时等 |

## 六、第一版最小可交付范围

第一版只锁定单台环控器产线检测，不做多设备复杂并发。必须完成：

1. `shared/constants.js` 建立 ATE 协议、寄存器、错误码、状态常量。
2. 后端完成 `AteFrameCodec`、`AteTcpClient`、`TestProtocol`、`TestManager`、`TestReportService`。
3. `DevicePool.js` 完成设备级 `runExclusive`；`PollingEngine.js` 完成测试期间跳过普通轮询和 12 秒冷却重连。
4. `server.js` 和 `WebSocketMgr.js` 完成 ATE WebSocket 消息与 `/api/test/*` API 接入。
5. `deviceStore.js` 增加 ATE 会话状态、命令发送、报告和日志状态。
6. `AteTest.vue` 从 mock 改为后端真实会话驱动，保留项目树、大表、历史报告、手动点检。
7. `App.vue` 补“自动化检定”菜单入口，建议使用 `ep-checked` 或 `ep-coordinate` 等质检类图标。
8. 覆盖 9 项基础硬件自检的 PASS、FAIL、STOP、RESET、ACK 超时。
9. 生成 JSON 和 HTML 报告，报告包含 SN、工单、固件版本、测试项、错误码、原始寄存器快照和排查建议。
10. 环控器固件端不新增开发，只联调既有 `ate_test_service`。

## 七、开发验收与测试建议

| 类别 | 验收命令或场景 | 通过标准 |
| :--- | :--- | :--- |
| 前端构建 | 在 `frontend` 执行 `npm run build` | 构建成功，无明显控制台错误 |
| 后端启动 | 在 `backend` 执行 `npm run start` 或 `npm run dev` | HTTP、WebSocket、OTA、ATE 模块初始化成功 |
| 协议单测 | 覆盖半包、粘包、坏 Magic、长度超限、非法 JSON | `AteFrameCodec` 不丢包、不误解析、不崩溃 |
| 真机 TCP JSON | 覆盖 `heartbeat`、`test.enter`、`test.exit`、`control.force_io`、`report`、ACK/error | UI 能显示状态和错误码 |
| Modbus ATE 区 | 读写 `0x8000`、读取进度和结果区 | 测试进度和结果与固件一致 |
| UI 尺寸 | 1366x768、1920x1080 | 无遮挡、无横向滚动、按钮和表格文本不溢出 |
| 安全释放 | 测试中停止、断线、页面刷新、后端重启 | 强制 IO 释放，报告记录异常 |
| 原功能回归 | 监控、继电器、OTA、设备管理 | 原 MaxClaw 功能不退化 |

## 八、风险与约束

| 风险 | 影响 | MaxClaw 侧处理建议 |
| :--- | :--- | :--- |
| 固件未烧录 ATE 测试版本 | 9001 不可达，测试无法启动 | UI 明确提示“设备未启用 ATE 测试固件”，禁止继续 |
| 多网卡选错本机 IP | DUT 回连或日志链路异常 | 必须支持 `LOCAL_IP` 手动指定，并在系统配置页展示 |
| 普通轮询与测试读写交错 | 自检状态错乱 | `DevicePool.runExclusive` + `PollingEngine.isDeviceUnderTest` |
| 半包/粘包处理不完整 | ACK 丢失或 JSON 解析失败 | `AteFrameCodec` 做充分单元测试 |
| Fanlogic 大包过大 | 固件 cJSON 解析失败或内存压力 | 按等级分包，等待 ACK 后继续 |
| 强制 IO 异常保持 | 现场安全风险 | `stop_test_request`、断线、心跳超时都触发 `test.exit` |
| 报告只在前端内存 | 断电丢失追溯 | 后端测试结束立即落盘 JSON |
| 当前 ATE 页面 mock 逻辑残留 | UI 显示与真实测试不一致 | `AteTest.vue` 只订阅 store，不自己模拟测试结论 |
| 图标和颜色依赖不规范 | 产线误操作或可访问性差 | 使用 Element Plus icons，状态不只靠颜色，还显示文字结论 |
