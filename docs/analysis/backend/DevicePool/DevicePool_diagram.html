【 DevicePool.js 】函数调用关系时序图
步骤 🧠 上层业务
server.js + PollingEngine.js 		⚙️ 连接池核心
DevicePool.js 		🔌 GD32 设备
Modbus TCP Client
🟢 初始化建池阶段
1
系统启动时 new DevicePool()
① 构造函数初始化两个 Map：devices 和 eventHandlers
—— 尚未连设备 ——
2
传入设备配置 { ip, port, unitId }
② addDevice(config)
拼出 key=ip:port:unitId，创建 ModbusRTU 客户端，状态置为 INIT
只是创建句柄，还没发任何报文
3
← 上层随时可查快照，不碰底层 Socket
③ getAllDevices() / getStatus() / getOnlineCount() →
🔄 正常通信阶段
4
上层请求 connect(key)
④ 状态先置为 CONNECTING，再调用 client.connectTCP()
发起 TCP 三次握手
5
← 回调/返回 true，并广播 ONLINE
⑤ 更新 status=CONNECTED、清零 timeoutCount、记录 lastSeen →
设备句柄进入可读写状态
6
轮询引擎调用 readHoldingRegisters()
⑥ 先校验 key 和连接状态，再发 Modbus 读报文
发送寄存器读请求
7
⑦ ← 返回寄存器数组，成功后重置超时计数并刷新 lastSeen
[258, 417, 0, 1, ...]
8
上层调用 writeRegister() / writeRegisters()
⑧ 校验连接后发 Modbus 写报文，成功时直接返回 true
心跳、继电器、OTA 都走这里
🛠️ 故障恢复与状态事件阶段
9
⑨ 读/写失败时：timeoutCount++
1~3 次只是软超时告警，异常继续往上抛
网络抖动 / 设备超时
10
← 对外发出 OFFLINE 事件，等下次业务层重连
⑩ 超过 3 次就 _resetConnection()：关闭句柄，状态置回 DISCONNECTED →
防止一直拿坏连接硬撑
11
← 手动断开时也会广播 OFFLINE
⑪ disconnect(key)：即使 close() 抛异常，也在 finally 里强制改状态
像“强制拔线后复位状态机”
12
← 已注册监听器才会真正收到事件
⑫ on(event, handler) / emit(event, ...args)
内建一个轻量事件总线，但当前项目里几乎没真正消费它
图例：
向设备发起连接 / 读取
向设备写寄存器
设备返回数据
状态事件 / 快照返回给上层

🔄 正常通信
