【 server.js 】函数调用关系时序图
步骤 🖥️ 配置 / 客户端
devices.json + Browser 		⚙️ 主入口
backend/server.js 		🔌 子模块 / 设备
Pool + WS + OTA + GD32
🟢 初始化启动阶段
1
读取 devices.json / .env
① loadConfig() 合并默认值、文件配置、环境变量覆盖
—— 尚未创建子模块 ——
2
② getLocalIP() 选出局域网 IP，给 OTA 下载地址和日志使用
3
③ new DevicePool() + addDevice()，把配置表注册成设备句柄
建立 key = ip:port:unitId
4
④ new OTAHandler() + startServer()，单独拉起固件下载 HTTP 服务
打开 OTA 端口，等待设备下载 SciGeneAI.rbl
5
⑤ new PollingEngine(devicePool, config.polling)，把轮询时钟和设备池绑在一起
轮询内核就绪，但还没开始跑
6
⑥ createServer(app) + new WebSocketManager({ server }) + start()
HTTP 和 WebSocket 共用同一个监听端口
7
⑦ 注册桥接逻辑
挂 onData/onClientMessage、挂静态资源、挂 OTA Router、挂 app.locals
子模块之间开始互相可见
8
⑧ server.listen()、connect()、pollingEngine.start()、startHeartbeat()
服务进入“持续运行”状态
🔄 运行期数据桥接阶段
9
⑨ ← 轮询子模块把 sensorData / 状态变化回调给 server.js
来自 PollingEngine 的异步事件
10
← 浏览器收到 sensor_data / device_status，页面实时刷新
⑩ wsManager.pushSensorData() / pushDeviceStatus() →
11
浏览器发来 relay_control / ota_start / get_devices
⑪ switch(message.type) 分发到 controlRelay() / triggerOTA() / sendToClient()
可能触发写寄存器或直接回包
12
浏览器访问 /api/devices、/api/health、/api/ota/*
⑫ Express 路由层负责同步查询和 OTA 接口入口
读取 devicePool / 调 OTAHandler / 调 PollingEngine
13
⑬ OTA 路由把“上传固件 / 查询状态 / 触发升级”继续委托给 OTAHandler 或 PollingEngine
一边存文件，一边给 GD32 下发升级命令
🧯 容错与退出阶段
14
⑭ process.on('uncaughtException' / 'unhandledRejection') 兜底记录未处理故障
主要是“记日志”，不负责自动恢复
停
终端发来 SIGINT（Ctrl+C）
⑮ 仅打印 [System] Shutting down... 后直接 process.exit(0)
当前实现没有逐个 stop() / disconnect()
